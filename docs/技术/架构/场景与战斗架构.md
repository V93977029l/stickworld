## 〇、文档目的

GDD 定义了"做什么"，本文档定义"怎么搭才不变屎山"。

P0 阶段三件事必须同时上线：
1. 火柴人行为 AI（特别是战斗 AI）
2. 小队级战斗（10-30 人）
3. 定居点建设基础逻辑

这三者高度耦合：工人建造时被袭击要切战斗 AI、战斗需要建筑作为掩体、建设需要建筑网格、战斗的编队直接复用组织系统的 L1 节点。本文档定义统一的场景骨架与模块边界，让三者各自开发但接口对齐。

## 一、设计原则（三条铁律）

### 1.1 逻辑层 ≠ 表现层

所有系统状态放在 `core/entities/` 的 State 对象里，**场景节点只是 State 的视图**。

- 火柴人 HP/位置/动画状态 → `StickmanState`
- 建筑血量/等级/占地 → `BuildingState`（新建）
- 战斗进度/参战双方 → `BattleState`（已有）
- 编制关系/指挥链 → `OrganizationState`（已有）

**场景卸载时 State 不丢**，挂载到新场景就能继续。这是支持"城镇就地变战场"、"快速旅行后状态延续"的基础。

### 1.2 常驻宿主 + 动态子场景

`GameRoot` 主场景常驻，村落/战场/室内都是动态加载的子场景。

- 环境、相机、UI、时间、输入分发跨场景保持
- 地图切换 = 替换 `WorldChunkHost` 的子节点

### 1.3 接口契约

模块间通信一律走 `api.gd` 或 EventBus。**任何 `get_node("../...")` 跨模块引用视为 Bug**。

依赖方向严格单向（见 §12 模块依赖图）。

## 二、GameRoot 主场景骨架

### 2.1 节点结构

```
GameRoot.tscn (常驻)
├── WorldClock              ← 接管 TimeManager，时间流速/暂停
├── EnvironmentSystem       ← 天空/天气/光照/震动（跨场景保持）
│   ├── SkyLayer            (极光、星星、太阳月亮 — Shader)
│   ├── WeatherLayer        (雨/雪/沙尘暴 — GPU 粒子)
│   ├── LightingLayer       (Environment + CanvasModulate)
│   └── GroundShakeRig      (整个世界父节点抖动)
├── CameraRig               ← 跟随玩家/小队/战略俯瞰
├── SceneLoader             ← chunk 流式加载/卸载
├── InputDispatcher         ← 模式路由（探索/建设/战斗/附身）
├── WorldChunkHost          ← 当前激活的地图实例挂载点
├── UIRoot                  ← 三层 UI 容器（见 §10）
└── BattleDirector          ← 多战场调度（城镇被袭->激活战斗实例）
```

### 2.3 启动流程

```
1. 项目入口 → SceneManager 加载 GameRoot.tscn
2. GameRoot._ready() 初始化所有常驻子系统
3. SceneLoader.load_map("village_start") 加载初始村落

   b. CameraRig.set_ground_y(map.ground_y)

5. InputDispatcher.set_mode(EXPLORE)
6. WorldClock.set_speed(1x)
```

### 2.4 CameraRig 详细设计（水平卷轴 + 自由镜头）

#### 2.4.1 设计原则

- **水平卷轴**：地图沿 X 轴展开，相机只在水平方向跟随角色
- **垂直显示范围限定**：相机 Y 不跟随角色 Y，而是由 `ground_y` 和 `ground_ratio` 计算，让地面线恒在屏幕下方某比例处（默认 0.4 = 2/5，可配置）
- **自由镜头**：玩家可手动拖动/边缘滚动观察地图任意位置，自动跟随暂停
- **缩放看特写**：缩放是为了观察战场细节，缩放时不保持地面比例

> **⚠️ 偏差说明（2026-07-18）**：v6 文档写"默认 1/3"，实际代码 `ground_ratio` 默认值为 **0.4（2/5）**（见 [camera_rig.gd:41](file:///f:/VSCode/game-2/stick-world/modules/world/scripts/camera_rig.gd) 与 [village_map.gd](file:///f:/VSCode/game-2/stick-world/modules/world/scripts/village_map.gd) `ground_ratio` 字段）。此处以代码为准。

#### 2.4.2 节点结构

> **⚠️ 偏差说明（2026-07-18）**：下图是设计目标结构。当前 [camera_rig.gd](file:///f:/VSCode/game-2/stick-world/modules/world/scripts/camera_rig.gd) 是单脚本实现，**未拆分** FollowController/ManualControl/ZoomController/ShakeRig 子节点，所有逻辑（跟随/手动/缩放/震屏）都在 CameraRig 脚本内。P0 阶段保持单脚本即可，后续若逻辑膨胀再拆分。

```
CameraRig (Camera2D)
├── FollowController (Node)     ← 自动跟随（1/4 区域触发，仅 X 轴）
├── ManualControl (Node)        ← 手动控制（拖动 + 边缘滚动）
├── ZoomController (Node)       ← 缩放（1.0x ~ 2.0x，鼠标锚点）
└── ShakeRig (Node)             ← 震屏
```

#### 2.4.3 垂直位置计算

```
camera_y = ground_y - viewport_height / zoom * (0.5 - ground_ratio)
```

- `ground_y`：地图地面线 Y 坐标（世界坐标系），由 MapInstance 提供
- `ground_ratio`：地面占屏幕高度比例（默认 0.33，Inspector 可改，每张地图可不同，如室内可能 0.5）
- `zoom_level`：当前缩放（1.0~2.0）
- 推导：地面线应在屏幕下方 `ground_ratio` 处（距底部 `ground_ratio * vp_h / zoom`），即 `ground_y = camera_y + vp_h/zoom * (0.5 - ground_ratio)`

**验证各比例：**
- `ratio=0.33`：地面在屏幕下方 1/3 处 ✓
- `ratio=0.5`：camera_y = ground_y，地面在屏幕正中 ✓
- `ratio=1.0`：camera_y = ground_y + vp_h/2，地面在屏幕顶部（满屏地面）✓

**计算时机：**
- 地图加载/切换时：用初始 zoom_level 计算一次 `camera_y`
- 缩放变化时：**不重算 camera_y**（缩放是看特写，以鼠标位置为锚点，地面比例自然变化）
- `ground_ratio` 变化时（Inspector 调整）：重算 `camera_y`

#### 2.4.4 水平跟随算法（1/4 区域触发）

> **⚠️ 文档/代码偏差修正（2026-07-18）**：v6 文档写"1/3 区域触发"，实际 [camera_rig.gd](file:///f:/VSCode/game-2/stick-world/modules/world/scripts/camera_rig.gd) 实现为 **1/4 区域触发**（`left_trigger = 1.0/4.0, right_trigger = 3.0/4.0`）。1/4 比 1/3 给中间区域更大游走空间，减少相机频繁移动。此处以代码为准。

将屏幕水平四等分，两侧各 1/4 为触发区，中间 1/2 为游走区：

```
┌──────────┬──────────────────────────┬──────────┐
│  左 1/4  │       中间 1/2           │  右 1/4  │
│(触发跟随)│      (角色可游走)         │(触发跟随)│
└──────────┴──────────────────────────┴──────────┘
```

**规则：**
1. 角色在中间 1/2 -> 相机不动
2. 角色进入左/右 1/4 -> 相机跟随，直到角色回到中间 1/2
3. **手动控制激活时**（拖动 / 边缘滚动 / 小地图跳转）-> 自动跟随暂停
4. **手动控制结束** -> 进入 5 秒冷却期，冷却结束后通过 1.5 秒缓动（`RETURN_SMOOTHING=2.5`）慢速弹回跟随位置，**不硬切**
5. `centered_mode = true` -> 强制角色到屏幕正中心，持续跟随（覆盖手动控制，禁用边缘滚动，松手即弹回）

**边界约束：** 相机**视野边缘**不超出 `[map_left, map_right]`（不是中心点 clamp）：
```
half_vp_w = viewport_width / (2 * zoom)
cam_x_min = map_left + half_vp_w
cam_x_max = map_right - half_vp_w
若 cam_x_min > cam_x_max（地图比视野窄）→ 居中显示
```

#### 2.4.5 手动控制

| 操作 | 行为 | 模式限制 |
|------|------|---------|
| 鼠标左键拖动 | 平移视野（水平方向） | EXPLORE/INDOOR |
| 鼠标移到屏幕左/右边缘 5% | 以 `edge_scroll_speed`（默认 400 px/s）水平滚动 | EXPLORE/INDOOR |
| 鼠标滚轮上 | 放大（zoom +0.1，上限 2.0），以鼠标位置为锚点 | 全模式 |
| 鼠标滚轮下 | 缩小（zoom -0.1，下限 1.0），以鼠标位置为锚点 | 全模式 |
| "居中"按钮（GlobalHUD） | 切换 `centered_mode` | 全模式 |
| 小地图点击（§10.4） | 相机跳转到点击位置（RTS 式），暂停自动跟随 | EXPLORE/INDOOR |

**手动控制与自动跟随的关系：**
- 任何手动控制激活期间，自动跟随暂停
- 手动控制结束后，自动跟随恢复正常规则（不拉回角色）
- `centered_mode` 是唯一能让相机主动拉回角色的方式

#### 2.4.6 缩放

- `zoom_level` 范围：**1.0（默认最小）~ 2.0（最大迫近）**
- 缩放以鼠标位置为锚点（鼠标指向的世界点保持屏幕位置不变）
- 缩放时 `camera_y` 不重算，地面占比自然变化（可能整个屏幕看地面厮杀）
- 缩放变化触发 `ViewportRect`（小地图视野框）更新

#### 2.4.7 公共 API

```gdscript
set_follow_target(node: Node2D)         # 设跟随目标（不强制中心）
clear_follow_target()
set_ground_y(y: float)                  # 设地面线（地图加载时调用）
set_ground_ratio(ratio: float)          # 设地面占比（Inspector 可改）
set_map_bounds(left: float, right: float)  # 设水平边界
set_zoom(level: float)                  # 1.0~2.0
get_zoom() -> float
shake(intensity: float)                 # 震屏
is_shaking() -> bool
set_centered_mode(enabled: bool)        # 居中跟随开关
get_viewport_rect() -> Rect2            # 当前视野世界矩形（供小地图用）
```

## 三、地图与 Chunk 系统

### 3.1 地图类型分类

| 类型 | 用途 | 特征 | 进入方式 |
|------|------|------|---------|
| `VillageMap` | 村落/定居点 | 水平卷轴 + 垂直显示范围限定（ground_y 约束，详见 §2.4.3），重复地面纹理 | 战略图进入 / 快速旅行 / 步行 |
| `BattlefieldMap` | 战场 | 更长更空旷，两端各有建筑/阵地，同样垂直显示范围限定 | 触发战斗进入 |
| `RoadMap` | 村落间道路 | 水平卷轴 + 垂直显示范围限定，可亲自步行 | 步行走出村落末端 |
| `IndoorMap` | 小型室内（铁匠铺） | 挂载在建筑内，墙面透明化，独立 ground_y（可能地面占比更大） | 走进建筑交互区 |
| `MegaInteriorMap` | 大型建筑（宫殿） | 独立完整场景，独立 ground_y | 传送切换 |
| `SlopeMap` | 山坡/阶梯地形 | **独立逻辑**：相机可倾斜旋转、ground_y 按 X 变化、台阶处需工程队搭爬梯，不复用 VillageMap 卷轴算法 | 特定区域传送 |

**关键**：
- `VillageMap` 与 `BattlefieldMap` 共用同一套 Chunk 系统，区别仅在数据配置（chunk 数量、地形空旷度、建筑密度）
- **所有地图类型都遵循"水平卷轴 + 垂直显示范围限定"原则**：相机 Y 不跟随角色 Y，由 `ground_y` + `ground_ratio` 计算（详见 §2.4.3）；火柴人脚部锁定在 `ground_y`（详见 §7.1）
- 每张地图可在 Inspector 调整 `ground_ratio`（默认 0.33，室内可能 0.5）

**`SlopeMap` 独立原则（重要）**：
山坡/阶梯/倾斜地形走独立 `SlopeMap` 类型，**不污染通用 VillageMap 架构**：
- `SlopeMap` 不复用 CameraRig 的水平卷轴算法，有自己的 `SlopeCameraRig`（可旋转/倾斜跟随）
- `SlopeMap` 有自己的 `SlopeStickmanController`（Y 跟随地形高度查表，不能自由上下移动）
- 火柴人在 SlopeMap 中只能沿斜坡前进/后退
- **台阶断崖**：火柴人无法直接跨越，需工程队搭建"爬梯"建筑（一种特殊建筑类型 `bld_ladder`）才能通过
- 普通地图（VillageMap/BattlefieldMap/RoadMap/IndoorMap/MegaInteriorMap）**保持简单**，不为山坡让步
- 实施时机：山坡地图实际开发时再实现，P0 阶段不涉及

### 3.2 Chunk 流式加载

每张地图由若干 **Chunk**（固定宽度的可游玩片段）拼接：

```
Village_A 地图：
  Chunk[0] | Chunk[1] | Chunk[2] | ... | Chunk[N]
  ← 旧     当前激活   预加载         末端 ->

玩家接近 Chunk[i] 末端触发器
  -> SceneLoader 预加载 Chunk[i+1]
  -> 玩家跨入 Chunk[i+1]
  -> SceneLoader 卸载 Chunk[i-1]（保留过渡缓冲）
```

**Chunk 规格**：
- 一个 Chunk ≈ 1~2 屏宽（约 1920~3840px）
- 每段含地面段 + 该段的建筑/装饰/触发器
- **跨 Chunk 边界**：地面纹理连续，建筑不能跨边界
- **Chunk 末端出口**：地图末端 Chunk 有"出口触发器"，跨入后 -> `SceneLoader` 加载相邻地图

> **⚠️ 实现状态（2026-07-18 审计）**：当前 Chunk 流式加载**仅为占位**。`SceneLoader.preload_chunk()` / `unload_chunk()` 只有 TODO 注释，直接 emit 信号不实际加载（见 [scene_loader.gd:236-246](file:///f:/VSCode/game-2/stick-world/modules/world/scripts/scene_loader.gd)）。当前每张地图是**整张一次性加载**的完整场景（`test_village_map.tscn` / `test_road_map.tscn` 等），未拆分为 Chunk。
>
> 阶段 0.8 的"多场景衔接"通过**地图末端出口触发器**（`ChunkTrigger` 节点，`target_map_id` 字段）实现**地图间切换**，而非地图内 Chunk 流式加载。这是两个不同层级：
> - **地图间切换**（已实现）：村落 A 末端 -> RoadMap -> 村落 B
> - **地图内 Chunk 流式**（未实现）：同一张地图内的分段加载
>
> P0 阶段单张地图足够小，整张加载性能可接受。真正需要 Chunk 流式时（大地图、长道路），再补全 `preload_chunk` / `unload_chunk` 实现。

### 3.4 MapInstance 节点结构

```
MapInstance (Node2D)
├── PlacementGrid                 ← 占地网格（见 §4.2）
├── TerrainLayer                  ← 地面纹理重复渲染
│   └── GroundPolygon             ← 地面多边形（顶点从 ground_y 开始向下）
├── GroundLine (Marker2D)         ← 🆕 地面线标记（y = ground_y，供调试可视化）
├── DecorationLayer               ← 装饰物（草、石头、墙）
├── BuildingHost                  ← 当前激活的建筑节点容器（运行时所有建筑实例都在此）
├── TerrainBuildings (Node2D)     ← 🆕 地形建筑（只读，随场景打包，不可拆除，见 §4.5）
├── InitialBuildingsList (Node)   ← 🆕 初始建筑数据列表（def_id + cell_x + width，见 §4.5）

│   └── BarrierRect[] (Area2D)    # 矩形障碍区域，火柴人不可穿越
├── BuildMaskLayer (Node2D)       ← 🆕 不可放建筑区域（大石头/山坡阶梯处，见 §4.2）
│   └── BuildMaskRect[] (ColorRect) # 设计时绘制，运行时注册到 PlacementGrid.blockage_mask
├── ForegroundLayer (Node2D)      ← 🆕 前景层（z_index=10，室内支柱/天花板，火柴人经过被遮挡）
├── EntityHost                    ← 当前激活的火柴人节点容器
├── ChunkTriggers                 ← 末端触发器
```

**z_index 层级约定**：

- DecorationLayer = 1
- BuildingHost = 2
- EntityHost = 3（火柴人）
- ForegroundLayer = 10（前景支柱最高）

## 四、建筑系统

### 4.1 资产分层（型号 vs 等级）

**核心规则**：建筑**型号**决定 `.tscn`，**等级**决定 `.tres` 数据。同型号不同等级 = 同场景 + 不同 Resource。

```
assets/buildings/
├── components/                    # 通用组件素材
│   ├── walls/                     (墙、屋顶、门的贴图集)
│   └── props/                     (桌椅、火炉等装饰)
└── templates/                     # 一种型号一个 .tscn
    ├── house/house.tscn
    ├── blacksmith/blacksmith.tscn
    ├── blacksmith/blacksmith_interior.tscn   # 内部场景
    ├── barracks/barracks.tscn
    └── ...

config/buildings/                  # 数据驱动（已有 buildings.tres 基础上扩展）
├── house_tier1.tres               # 等级配置：贴图变体、占地格、容量、HP
├── house_tier2.tres
└── ...
```

**贴图尺寸 vs 占地尺寸（解耦）**：
- **占地尺寸**（`footprint_w`/`footprint_h`）= 格子数 × 32px，决定 PlacementGrid 占用
- **贴图尺寸**任意，可以比占地大或小
- 例：一棵树占地 1×1（32×32px，树根），贴图 96×192px（树冠很宽很高），Sprite2D 用 `offset = (-32, -160)` 让树冠向上向左展开
- 贴图超出占地范围不冲突，PlacementGrid 只校验占地格子

**程序化建筑扩展点（架构已就绪，P0 不实现）**：
建筑场景根节点可挂 `procedural_building.gd` 脚本，`_ready()` 中根据 `variant_seed` + `terrain_context`（地形类型、周围建筑、海拔）调用 `RandomNumGenerator` 选择 sprite 子集拼接。
- 建筑数据表（xlsx）新增字段：`procedural: bool`、`variant_seed: int`、`terrain_adaptive: bool`
- 地标建筑：地图设计时放置 `procedural=true` 实例，每个 `variant_seed` 不同 -> 每个地区天然不一样
- 玩家建筑：建造时传入 `terrain_context`，建筑脚本根据 context 生成不同造型
- 当前建筑场景文件设计为"可挂脚本"就为以后留好了口子，无需改架构

### 4.2 占地网格（32px 高精度）

```
modules/world/placement_grid/
├── placement_grid.gd              # 网格管理
├── grid_cell.gd                   # 单元格
└── placement_validator.gd         # 校验是否可建
```

每个 Map 持有一个 `PlacementGrid`：
- `cell_size = 32px`
- 每个建筑占据 N×M 个 cell（小屋 2×2、兵营 3×3、宫殿 8×6）
- **一块地只能建一个建筑** = cell 占用检查
- 选址预览 = ghost 建筑半透明 + 网格高亮（绿=可建/红=冲突）

**BuildMask 不可放建筑区域（地形限制）**：

PlacementGrid 新增 `blockage_mask: PackedByteArray`（与 grid 同尺寸），标记"地形上不可放建筑"的格子：

```gdscript
# PlacementGrid 新增字段
var blockage_mask: PackedByteArray  # 1=不可放建筑（大石头/山坡阶梯处/装饰物占地）

func is_blocked(x: int, y: int) -> bool:
    return blockage_mask[y * grid_width + x] == 1

func is_occupied(x: int, y: int) -> bool:
    # 原有建筑占用 OR blockage_mask = 1
    return _cells[y * grid_width + x] != "" or is_blocked(x, y)
```

**设计时操作**：地图场景的 `BuildMaskLayer` 节点下加 `ColorRect`（红色半透明，仅设计时可见），运行时 `_ready()` 读取这些 ColorRect 的位置尺寸，注册到 `PlacementGrid.blockage_mask`。

**与"路障建筑"方案对比（已否决）**：
- 路障建筑方案被否决，原因：污染建筑系统、调试显示乱、拆除逻辑需特殊处理
- BuildMask 是纯数据，不会误删、不会出现在建筑列表、与 PlacementValidator 自然集成
- 玩家尝试在 BuildMask 区域放建筑 -> PlacementValidator 返回冲突，提示与"建筑占用"相同的错误信息

**调试显示**：DebugOverlay 开启时，BuildMask 格子显示为红色半透明（详见 §10.5）。

### 4.3 建筑节点结构

```
Building (Node2D)                   ← 运行时实体
├── Footprint (CollisionShape2D)    ← 占地碰撞，对应 PlacementGrid 占用
├── Exterior (Node2D)               ← 屋顶 + 前墙（默认可见）
│   └── WallFront (Sprite2D)        ← 进入交互区时 modulate.a → 0.3
├── Interior (Node2D)               ← 内部空间 + NPC + 工作位
│   └── 默认 visible=false，进入交互区时 visible=true
├── InteractionZone (Area2D)        ← 玩家进入触发"室内模式"
├── PassageBarrier (Area2D)         ← 🆕 建筑级通行障碍（见 §7.1.3）
│   └── CollisionShape2D[]          # 矩形障碍区域，火柴人不可穿越
├── HealthComponent (Node)          ← HP/伤害
├── WorkSlot (Node2D)[]             ← 工作位（工人 AI 寻路目标）
└── CoverMarker (Node2D)[]          ← 掩体标记（战斗 AI 查询）
```

**建筑级 PassageBarrier（建筑本体不可通行区域）**：

建筑场景设计时，在 `PassageBarrier` 节点下加 `CollisionShape2D`（RectangleShape2D），划定建筑本体哪些区域火柴人不可穿越。与地图级 `WalkBarrier` 共用同一套碰撞查询（详见 §7.1.3）。

**典型用例**：

| 建筑类型 | 通行规则 | PassageBarrier 配置 |
|---------|---------|---------------------|
| 雕像（小） | 角色从上下两侧都可绕过 | 本体一小块矩形（仅雕像基座） |
| 房屋 | 角色只能从房屋下方通过 | 覆盖房屋上半部分（墙体），下方留空 |
| 兵营 | 角色可从下方过，正门交互进入 | 覆盖建筑上半部分 + 左右两侧 |
| 矿山（地形建筑） | 角色完全不能穿过 | 整个占地范围全覆盖 |
| 树木（装饰） | 角色可绕过 | 仅树根一小块 |

**设计原则**：
- PassageBarrier 形状可与占地不同（占地是水平条带宽度，PassageBarrier 是物理碰撞）
- 例：房屋占地 3 格宽（96px），PassageBarrier 只覆盖上半部分 96×64px，下方 96×32px 留给角色通行
- 设计时在建筑场景编辑器直接绘制 CollisionShape2D，所见即所得
- 调试显示：DebugOverlay 开启时，建筑 PassageBarrier 显示为紫色半透明（与地图级 WalkBarrier 蓝色区分，详见 §10.5）

### 4.5 建筑三层架构（地形 / 初始 / 玩家）

游戏内有三种建筑来源，它们**和平共存**，通过同一套独立场景文件加载，但存储位置和可修改性不同。

#### 4.5.1 三层对比

| 层 | 存储格式 | 可破坏 | 加载时机 | 示例 |
|---|---|---|---|---|
| **地形建筑层** | 地图场景 `.tscn` 中 `TerrainBuildings` 节点 | 否 | 随场景实例化 | 矿山、遗迹、大石头 |
| **初始建筑层** | 地图场景 `.tscn` 中 `InitialBuildingsList` 列表（def_id + cell_x + width） | 是 | 地图加载后写入存档 | 关卡设计放的民居 |
| **玩家建筑层** | 存档 `.json` | 是 | 存档加载时实例化 | 玩家新建的农场 |

#### 4.5.2 为什么不会冲突

```

1. SceneLoader 实例化 map.tscn
   └── TerrainBuildings 下的建筑自动随场景创建（静态，注册到 PlacementGrid，标记 is_terrain=true）
2. VillageMap._ready() 读取 InitialBuildingsList
   └── 首次进入该地图：将列表写入存档 JSON 的 buildings 数组
   └── 非首次：跳过（存档已有）
3. SaveManager 读取存档 JSON
   └── 对每个动态建筑实例（初始 + 玩家）：
       ├── 加载对应场景文件
       ├── 实例化后 add_child 到 BuildingHost
       └── 注册到 PlacementGrid
```

### 5.2 小型建筑 - 墙面透明化（不传送）

#### 5.2.1 透明化状态机

每栋建筑独立维护透明化状态，互不影响（多建筑重叠见 §5.5）：

```
状态 TRANSPARENT_OFF（默认）：
  Exterior.visible = true
  WallFront.modulate.a = 1.0
  Interior.visible = false

  ── 玩家进入 InteractionZone ──>

状态 TRANSPARENT_ON：
  Exterior.visible = true（屋顶仍可见，保持空间感）
  WallFront.modulate.a = 0.3（前墙半透明，Tween 渐变 0.2s）
  Interior.visible = true（内部显示）

  ── 玩家离开 InteractionZone ──>

状态 TRANSPARENT_OFF（反向恢复，Tween 渐变 0.2s）
```

**关键**：
- 只调 `WallFront` 的 alpha，**不调整个 `Exterior`**（屋顶始终可见，保持空间感）
- 渐变用 `Tween` 实现 0.2s 过渡，避免硬切
- `Interior` 含工作位 NPC、打铁动画、内部装饰，透明化时才渲染

#### 5.2.2 InteractionZone 触发机制

`InteractionZone` 是 Building 的 `Area2D` 子节点（见 §5.3），检测**玩家附身实体**进入：

```gdscript
# Building._on_interaction_zone_body_entered(body)
func _on_interaction_zone_body_entered(body: Node2D) -> void:
    if not _is_player_entity(body):
        return  # 非玩家实体（NPC）不触发，避免全图半透明
    _set_transparent(true)
    EventBus.interior_entered.emit(get_instance_id())
    # 首个进入的建筑触发模式切换
    if _input_dispatcher.get_mode() != PlayerControlAPI.Mode.INDOOR:
        _input_dispatcher.enter_indoor_mode()

func _on_interaction_zone_body_exited(body: Node2D) -> void:
    if not _is_player_entity(body):
        return
    _set_transparent(false)
    EventBus.interior_exited.emit(get_instance_id())
    # 仅当所有交互区都不含玩家时才退回 EXPLORE（见 §5.5）
    _maybe_exit_indoor_mode()
```

**`_is_player_entity` 判定**：检查 body 是否为当前附身实体（`PossessionInterface.get_possessed_entity()`）。非附身 NPC 走进建筑**不触发**透明化，避免视觉混乱。

#### 5.2.3 优点

- 室外正常显示，玩家能看见"师傅在打铁"
- 进入是连续的，无传送感
- 多个建筑可同时透明化（玩家跨越多个交互区时，见 §5.5）

### 5.4 INDOOR 模式行为契约

#### 5.4.1 与 EXPLORE 的关系

INDOOR 是 EXPLORE 的**子模式**，玩家仍控制角色移动（WASD），区别仅在：

| 维度 | EXPLORE | INDOOR |
|------|---------|--------|
| 角色移动 | WASD | WASD（相同） |
| 相机跟随 | 水平卷轴 + 1/4 区域 | 水平卷轴 + 1/4 区域（**不变**） |
| 相机 ground_y/ground_ratio | 地图级 | 地图级（**不变**，不用建筑级） |
| 相机手动控制 | 拖动 + 边缘滚动 + 缩放 | 拖动 + 边缘滚动 + 缩放（相同） |
| UI | VillagePanel | VillagePanel（**不变**） |
| 输入路由 | ExploreHandler | ExploreHandler（**复用**，不新建 IndoorHandler） |

**关键决策**：INDOOR 模式**不新建 handler**，复用 ExploreHandler。INDOOR 仅作为"当前在室内"的标记，供 UI/音效系统查询（如室内混响、UI 提示"按 E 交互"）。

> **为什么不用建筑级 ground_ratio**：建筑内地面 Y 与地图地面 Y 相同（建筑贴地建造），无需独立相机参数。若未来室内有不同高度（如二楼），再考虑建筑级相机偏移。

#### 5.4.2 模式切换规则

```
EXPLORE ──进入 InteractionZone──> INDOOR
INDOOR ──离开所有 InteractionZone──> EXPLORE
INDOOR ──进入战斗（城镇被袭）──> BATTLE
INDOOR ──打开战略图──> UI（战略图关闭后恢复 INDOOR）
```

**禁止从 INDOOR 直接到 BUILD/POSSESS**：建设需先离开交互区回 EXPLORE；附身需通过 BATTLE 模式。

### 5.5 多建筑重叠与边界情况

#### 5.5.1 多建筑同时透明化

玩家可能同时处于多个建筑的 InteractionZone 内（如两个紧邻的铁匠铺）：

- **每个建筑独立维护透明化状态**：进入各自 InteractionZone 各自变透明，互不影响
- **INDOOR 模式只切换一次**：首个进入的建筑触发 `enter_indoor_mode()`，后续建筑不再重复切换
- **退出时检查全局**：离开某建筑 InteractionZone 时，检查是否还在其他建筑的 InteractionZone 内。**仅当所有交互区都不含玩家时**才 `exit_to_explore()`

```gdscript
# Building._maybe_exit_indoor_mode()
func _maybe_exit_indoor_mode() -> void:
    if _input_dispatcher.get_mode() != PlayerControlAPI.Mode.INDOOR:
        return
    # 遍历当前地图所有建筑，检查是否还有任意 InteractionZone 含玩家
    var any_inside := false
    for building in _building_host.get_children():
        if building is Building and building._is_player_inside_interaction_zone():
            any_inside = true
            break
    if not any_inside:
        _input_dispatcher.exit_to_explore()
```

#### 5.5.2 边界情况

| 场景 | 处理 |
|------|------|
| 玩家附身时进入交互区 | 正常触发透明化（附身实体就是玩家实体） |
| 玩家取消附身时在室内 | 透明化保持，模式回 EXPLORE（附身->EXPLORE 不强制退出 INDOOR） |
| 建造中建筑（UNDER_CONSTRUCTION） | InteractionZone 不激活（未完工无内部可看） |
| 被破坏建筑（DESTROYED） | InteractionZone 不激活 |
| 战斗中进入交互区 | 透明化仍生效，但模式切到 BATTLE（BATTLE 优先级高于 INDOOR） |
| 传送进入 MegaInteriorMap 后 | 新地图是独立场景，无 InteractionZone 概念，模式为 INDOOR（新地图默认） |

### 5.6 大型建筑 - 传送切换（MegaInteriorMap）

### 6.1 三种旅行方式

| 方式 | 触发 | 实现 | 限制 |
|------|------|------|------|
| 步行 | 玩家走到村落末端出口 | 加载 RoadMap → 末端走出 → 加载下一村落 | 无限制 |
| 快速旅行 | 大世界地图选目标 | 显示"行进中"过场 + 消耗游戏内时间 → 加载目标 | 战斗中禁止 |

### 6.2 步行流程

```
Village_A 末端出口触发器
  → SceneLoader.load_map("road_A_to_B")
  → RoadMap 加载，玩家从左端进入
  → 玩家走完 RoadMap
  → 末端出口触发器
  → SceneLoader.load_map("village_B")
```

`RoadMap` 也支持 Chunk 流式加载，可长可短。后期可加入路上事件（伏击、商人、流民）。

### 6.3 快速旅行流程

```
玩家打开战略图 (Tab)
  → world_map 模块（战略图模块）显示已探索村落/城市
  → 玩家选择目标
  → 校验：当前不在战斗中？目标已探索？

  → SceneLoader.load_map(target_id)
  → 玩家出现在目标村落的入口
```

**设定合理性**："快速旅行"在游戏内解释为"主角走过去了"，只是省略了过程。消耗的游戏内时间按距离计算。

### 6.4 旅行封锁条件

- 当前 Map 挂载了 active `BattleInstance` → 禁止
- 玩家正在被攻击（HP < 阈值且最近受击 < 5s）→ 禁止
- 目标未探索 → 不可选

## 七、火柴人行为 AI

### 7.1 StickmanEntity 节点结构（在已有 StickmanRig 之外包一层）

```
StickmanEntity (CharacterBody2D)        ← 🆕 物理+碰撞
├── StickmanRig (Skeleton2D)            ← ✅ 已有，纯渲染骨架
├── Hitbox (Area2D)                     ← 受击判定
├── WeaponMount (Node2D)                ← 武器挂载
├── HealthComponent (Node)              ← HP/士气
├── AIController (Node)                 ← 🆕 决策大脑
│   └── BehaviorStateMachine
└── PossessionInterface (Node)          ← 🆕 玩家附身接口
```

**关键**：`StickmanRig` 不动（已实现的 IK/动画保留），外层包一个 `CharacterBody2D` 承担物理与移动。AI 通过 `rig.play("walk")` + `rig.set_anim_speed(...)` 驱动渲染，通过 `CharacterBody2D.velocity` 驱动物理。

#### 7.1.1 地面约束（Y 范围 + X 边界）

火柴人可在地面区域内上下左右自由移动（不是锁死在 ground_y 一条线）：

```gdscript
# StickmanEntity 字段
var ground_y: float = 300.0       # 地面顶部 Y（由 MapInstance.spawn_entity 注入）
var ground_bottom: float = 1024.0 # 地面底部 Y（火柴人可走区域底部）
var map_left: float = 0.0         # X 活动范围左边界
var map_right: float = 2048.0     # X 活动范围右边界
var foot_offset: float = 30.0     # 脚部到节点原点的偏移（CollisionShape2D 半高）

func _physics_process(delta):
    # ... 输入处理（WASD 上下左右）...
    move_and_slide()
    # Y 范围约束：脚部保持在 [ground_y, ground_bottom] 内
    var y_min: float = ground_y - foot_offset
    var y_max: float = ground_bottom - foot_offset
    global_position.y = clampf(global_position.y, y_min, y_max)
    # X 边界约束
    global_position.x = clampf(global_position.x, map_left, map_right)
```

**生成位置：** 火柴人生成 Y = `ground_y + (ground_bottom - ground_y) * 0.5`（地面垂直范围偏中心）。

**未来扩展（非 P0）：**
- 跳跃：`velocity.y` 临时非零，落地后重新受 Y 范围约束
- 飞行单位：`position.y` 可超出 `ground_y`（向上飞），但仍受 `ground_bottom` 约束
- 地形高度变化：地图不同 X 位置 `ground_y` 不同（查表），火柴人 Y 跟随地形

#### 7.1.2 通行障碍系统（WalkBarrier + PassageBarrier）

火柴人移动除了受 `ground_y`/`ground_bottom`/`map_left`/`map_right` 矩形约束外，还受**两级透明障碍**约束：

**地图级 WalkBarrier（悬崖/高楼边缘）**：
- 位于 `MapInstance.WalkBarrier` 节点下，含若干 `Area2D` + `CollisionShape2D`（矩形）
- 设计时在地图场景中绘制，覆盖悬崖边缘/高楼边界/断崖
- 运行时不可见，调试模式显示为**蓝色半透明矩形**（详见 §10.5）
- 适用场景：悬崖边缘、高楼堡垒边缘、断崖、任何"角色不能御空而行"的边界
- **不适用于山坡**（山坡走 `SlopeMap` 独立逻辑，详见 §3.1）

**建筑级 PassageBarrier（建筑本体不可通行）**：
- 位于每个建筑场景的 `PassageBarrier` 子节点下（详见 §4.3）
- 设计师在建筑场景中绘制，划定建筑本体哪些区域不可通行
- 运行时不可见，调试模式显示为**紫色半透明矩形**（与地图级蓝色区分）
- 典型用例：雕像上下可过、房屋只能从下方过、矿山完全不可过

**火柴人碰撞查询**：

```gdscript
# StickmanEntity._physics_process
var _last_valid_position: Vector2

func _physics_process(delta):
    # ... 输入处理 + move_and_slide() + Y/X 矩形约束 ...

    # 🆕 通行障碍检测：若进入任何 WalkBarrier / PassageBarrier 区域，回退到上一帧位置
    if _is_in_passage_barrier():
        global_position = _last_valid_position
        velocity = Vector2.ZERO  # 撞墙停止
    else:
        _last_valid_position = global_position

func _is_in_passage_barrier() -> bool:
    # 查询当前 MapInstance.WalkBarrier 下所有 Area2D
    # 查询 BuildingHost 下所有建筑的 PassageBarrier Area2D
    # 用 PhysicsDirectSpaceState2D.intersect_point 或 Area2D.overlaps_body 检测
    # 实现细节：见 modules/units/scripts/stickman_entity.gd
    ...
```

**与 SlopeMap 的关系**：
- WalkBarrier/PassageBarrier 是"矩形区域阻挡"的轻量方案，适用于 VillageMap 等水平卷轴地图
- SlopeMap 走独立逻辑，不用 WalkBarrier（坡面是连续的 Y 变化，不是矩形阻挡）

```
┌──────────────────────────────────────┐
```

- 高优先级指令覆盖中低优先级

### 7.4 小兵步枪式灵动性 — 两层实现

**第一层：行为层概率钩子**

每个战斗行为内置可配置概率：
```gdscript
# behavior_attack.gd 示意
@export var prob_aggressive_push: float = 0.05  # 擅自冲锋概率
@export var prob_hesitate: float = 0.03          # 犹豫概率

func update(delta):
    if _is_at_disadvantage() and randf() < prob_hesitate:
        _enter_hesitate_substate()
    elif _enemy_exposed() and randf() < prob_aggressive_push:
        _push_forward()
```

**第二层：战场导演情绪标签**

`battle_ai_director.gd` 周期性（每 2~5s）给单位打"情绪标签"：
- `HESITANT` — 犹豫（命中率-30%、移动减速）
- `EXCITED` — 亢奋（追击倾向+50%、忽视指令概率+10%）
- `PANICKED` — 恐慌（找掩体优先级最高、可能溃逃）
- `STEADY` — 稳定（默认）

情绪概率受：指挥官能力、部队士气、文化传统、自主决策权限影响。

### 8.5 指挥链延迟

命令从 L(n) 下达到 L(n-1) 有延迟，模拟传令时间：
- 延迟 = `base_delay × tier_diff × commander_efficiency_modifier`
- 默认 `base_delay = 2s`，每跨一层 +2s
- 指挥官能力高 → 延迟减半
- 玩家附身该层级指挥官时 → 延迟为 0（直接指挥）

### 9.3 工程量系统（非固定建造时间）

```gdscript
remaining_work = base_work  # 例如铁匠铺 = 100 工时

# 每帧推进
func _process(delta):
    var worker_count = _get_assigned_workers().size()
    var efficiency = _get_crew_efficiency()  # 受组织水平影响
    var work_done = worker_count * efficiency * delta
    remaining_work -= work_done
    if remaining_work <= 0:
        complete()
```

#### 10.5.2 启动图例（常驻小字）

游戏启动时在屏幕左上角显示**小字号**图例，**不淡出**，常驻显示（直到玩家按 F3 关闭调试模式）。图例内容：

```
[F3] 调试
绿格=占地  红格=不可建  蓝区=地图障碍  紫区=建筑障碍
黄线=地面线  青线=地面底  白框=建筑  紫框=Chunk触发
```

图例字号小（约 10pt），半透明（alpha=0.6），不遮挡游戏画面。F3 切换调试模式时图例同步显示/隐藏。

#### 10.5.3 显示内容明细

| 元素 | 颜色 | 来源 | 说明 |
|------|------|------|------|
| PlacementGrid 占用格 | 绿色半透明 | `PlacementGrid._cells` | 已被建筑占用的格子 |
| BuildMask 不可放建筑格 | 红色半透明 | `PlacementGrid.blockage_mask` | 地形限制不可放建筑 |
| WalkBarrier 地图障碍 | 蓝色半透明 | `MapInstance.WalkBarrier` 下 Area2D | 悬崖/高楼边缘，火柴人不可穿越 |
| PassageBarrier 建筑障碍 | 紫色半透明 | 各建筑 `PassageBarrier` Area2D | 建筑本体不可通行区域 |
| ground_y 地面线 | 黄色线 | `MapInstance.ground_y` | 火柴人可走区域顶部 |
| ground_bottom 地面底线 | 青色线 | `MapInstance.ground_bottom` | 火柴人可走区域底部 |
| 建筑边界框 | 白色边框 | `BuildingHost` 子节点 | 每个建筑的 Footprint 矩形 |
| Chunk 触发器范围 | 紫色矩形边框 | `MapInstance.ChunkTriggers` | 流式加载触发区域 |
| 火柴人状态文字 | 白色文字 | `EntityHost` 下 StickmanEntity | 速度/动画/朝向/坐标 |
| FPS / 实体数 | 白色文字 | 引擎 | 屏幕左下角 |

#### 10.5.4 交互

| 按键 | 行为 |
|------|------|
| F3 | 切换调试覆盖层显示/隐藏（记忆到 `settings.cfg`） |
| F1（可选） | 仅切换图例显示 |

#### 10.5.5 实现要点

- `DebugOverlay` 是 `CanvasLayer`，`z_index` 最高（高于游戏画面和 UI）
- 用 `_draw()` 在 Control 上绘制（不是 Node2D，避免相机移动影响坐标）
- 调试数据通过 `DebugApi` 收集，各模块注册自己的调试信息提供者
- 启动时调用 `debug_legend.show_legend()`，图例常驻不淡出
- 性能：调试绘制器按需启用，关闭调试模式时所有 `_draw` 跳过
- 世界坐标到屏幕坐标的转换：`world_to_screen(pos) = (pos - camera.global_position) * effective_zoom + viewport_size * 0.5`

#### 10.5.6 公共 API

```gdscript
# DebugApi（autoload 单例）
register_drawer(name: String, drawer: Callable)   # 各模块注册绘制器
unregister_drawer(name: String)
toggle_visibility()                                # F3 切换
is_visible() -> bool
show_legend()                                      # 显示图例（常驻）
hide_legend()
```

#### 10.5.7 各模块注册绘制器

| 模块 | 注册的绘制器 | 触发时机 |
|------|------------|---------|
| `world` | grid_drawer / barrier_drawer / ground_line_drawer / chunk_trigger_drawer | 地图加载时 |
| `buildings` | building_drawer | 建筑增减时 |
| `units` | entity_state_drawer | 单位生成时 |

### 11.1 模块结构

```

├── weather_system.gd          # 天气：雨/雪/沙尘暴

├── lighting_system.gd         # 光照：随时间变化

```

### 11.2 跨场景保持

环境系统是 `GameRoot` 的常驻子节点。地图切换时只是切换 `WorldChunkHost` 的子节点，**环境继续工作**。

环境参数（天气、时间、地理位置）挂在 `WorldState`（Autoload）上，不挂场景。

### 11.3 关键技术点

| 效果 | 实现方式 |
|------|---------|
| 极光 | `Shader` + 噪波纹理 + 时间动画 |
| 星星 | `GPUParticles2D` 或 Shader 顶点 |
| 下雨 | `GPUParticles2D` + 风向参数 |
| 沙尘暴 | `GPUParticles2D` + 屏幕色染 + 能见度衰减 |
| 地面震动 | `GroundShakeRig` 父节点 `position` 抖动（参考相机抖动） |
| 光照变化 | `CanvasModulate.color` 按 `time_of_day` 插值 |
| 生物群落基调 | `biome_system` 提供参数，`sky`/`weather`/`ground` 各自读取 |

- `combat` 不直接调 `construction`，通过 EventBus 通知（如建筑被破坏）
- 跨模块引用 → 走 `api.gd` 或 EventBus

### P0-1 🔴 PossessionInterface / PossessPanel 未装配（严重，功能不可用）

- **根因**：[game_root.gd](file:///f:/VSCode/game-2/stick-world/modules/world/scripts/game_root.gd) 在某次重构中**删除了** `_setup_possession_interface()` / `get_possession_interface()` / `_setup_possess_panel()` / `get_possess_panel()` 方法，但调用方（[battle_panel.gd:335](file:///f:/VSCode/game-2/stick-world/modules/ui/scripts/battle_panel.gd)、[possess_panel.gd:29](file:///f:/VSCode/game-2/stick-world/modules/ui/scripts/possess_panel.gd)、[test_stage_07.gd:98](file:///f:/VSCode/game-2/stick-world/tests/test_stage_07.gd)）未同步清理。`InputDispatcher` 也未注册 `POSSESS` handler。

  2. 在 `_ready()` 装配序列中调用
  3. `InputDispatcher.register_handler(InputDispatcher.Mode.POSSESS, possession_interface)` 注册
  4. 重新运行 test_stage_07 验证
  5. 删除或更新过期的 `test_stage_07_result.txt`
- **优先级**：**阻塞阶段 0.9 及后续所有附身相关功能**
