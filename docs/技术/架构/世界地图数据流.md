## 〇、本文档要解决的问题

程序化世界生成（Python 工具链）和战略图模块（`modules/world_map/`）分别独立设计，在「产物交付」这一步出现断点：

- 生成端产出的是 PNG 掩码 + numpy 高度场 + 将来的多边形 JSON
- 消费端期望的是 `region_mask_texture`（像素 RGB 编码 region_id 的索引图）+ `RegionDefinition` 列表

本文档定义两者之间的**转换链路、数据契约、导入器接口**，作为后续编码的真相源。

## 一、术语对照（先统一语言）

### 1.1 两类地图的本质区分

游戏内同时存在两类「地图」，过去文档混用「地图」一词导致歧义。本文档明确区分：

| 术语 | 英文 | 定义 | 模块 | 玩家是否在其中 |
|------|------|------|------|--------------|
| **战略图** | Strategic Map | 鸟瞰多边形领土的「看地图」，用于指挥、缩放、外交、查看版图 | `modules/world_map/` | **否**，玩家在场景图里，战略图是「右上角小地图」放大版 |
| **场景图** | Scene Map | 玩家实际所在的卷轴/独立场景，火柴人在其中真实移动 | `modules/world/` 等 | **是**，玩家操控的单位在场景图内 |

> **类比**：P社游戏中「政治地图模式」（战略图）vs「打开一个省份看战斗」（场景图）。战略图永远不挂载 StickmanEntity，只显示色块/图标/战线。

### 1.2 GDD §6.7 的六类地图全部是场景图

GDD §6.7 列举的村落/战场/道路/室内/大建筑/山坡六类，**全部是场景图**。战略图不在该表中，是独立的第七类（或者说，是与六类正交的「元视图」）。

### 1.3 三级层级与 GDD 五层级的对应

程序化生成文档的 L3/L2/L1 与 GDD §4.2 的五层级管理深度是同一套层级的两种视角：

| 程序化文档术语 | GDD §4.2 术语 | 行政语境 | 战略图缩放级别 | 对应战略图数据粒度 |
|--------------|--------------|----------|--------------|------------------|
| **L3 大世界** | 第5层 帝国级 | 整个已知世界 | 最远（看全大陆） | `ContinentData`：大陆轮廓 + 地区多边形 |
| **L2 地区** | 第4层 行省级 | 区域/行省 | 中（看一个地区） | `RegionData`：地区多边形 + 地块多边形 |
| **L1 地块** | 第3层 城市/郡县级 | 城市/郡县 | 近（看一个地块） | `TileData`：地块多边形 + 聚落点位 |
| —（场景图） | 第2层 镇/村级 | 镇/村 | 进入聚落即切到场景图 | `VillageMap` 等六类场景图 |
| —（玩家小队） | 第1层 个体/班组 | 排/班 | 场景图内附身操控 | `StickmanEntity` |

> **关键**：战略图只覆盖 GDD 第5/4/3 层级。第2层（聚落）和第1层（玩家小队）属于场景图范畴，不在战略图模块内。从战略图「点击聚落」即触发 `SceneLoader.load_map()` 切换到场景图。

## 二、生成端产物清单

来自 [`程序化世界生成.md`](../../设计/系统/08-程序化世界生成.md) §十二的数据流，按交付阶段分类。

### 2.1 阶段 A：世界模板（开发期生成，随包发布，永不再变）

| 产物 | 格式 | 来源 | 内容 |
|------|------|------|------|
| `locked_continent_8192.png` | PNG（L 模式二值） | `fractal_continent.py` | 大陆掩码，1=陆地 0=海洋，8192×8192 |
| `locked_continent_8192.png.npy` | numpy uint8 | 同上 | 同上的数组形式，便于 Python 处理 |
| `fractal_heightmap_8192.npy` | numpy float32 | 同上 | 高度场，值域 0-1，含外海/内湖/山脉 |
| `preview_fractal.png` | PNG RGB | 同上 | 高度场彩色预览（8 色分级），人眼查看用 |
| `world_map_l3.png` | PNG RGB | `world_map.py` | L3 群系+河流渲染图（当前阶段产物） |
| **`continent_data.json`** | JSON | **待实现** | L3 大陆元数据：大陆名、地区列表、地区多边形顶点、地区邻接图 |
| **`regions_data.json`** | JSON | **待实现** | L2 地区元数据：每个地区内的地块多边形、地块邻接图、地块群系 |
| **`tiles_data.json`** | JSON | **待实现** | L1 地块元数据：每个地块内的聚落列表（级别/产业/位置/规模）、资源禀赋、河流裁剪 |
| **`political_data.json`** | JSON | **待实现** | 政权数据：政权/联盟/归属/都城（程序化文档 §15） |

> 标注「待实现」的三份 JSON 是桥接文档要求生成端补产的，目前 `fractal_continent.py` 只产出 PNG/npy，未产出结构化多边形数据。**这是生成端的工作项，不是消费端的工作项**。

### 2.2 阶段 B：每局运行时扰动（仅规模）

| 产物 | 格式 | 来源 | 内容 |
|------|------|------|------|
| `run_data.tres` | Godot Resource | `RunPerturbator`（运行时） | 仅聚落规模 ±15% 扰动，其他全部不变 |

### 2.3 阶段 C：按需可玩地图（玩家进入聚落时）

不属于本数据流——这是场景图侧的事，由 `SettlementGenerator` 产 `MapDefinition.tres` 喂 `SceneLoader`。详见 [`场景与战斗架构.md`](场景与战斗架构.md) §三。

## 三、消费端输入清单

`modules/world_map/` 的战略图渲染需要以下输入（来自 [`战略图架构.md`](战略图架构.md) 的重新设计）。

### 3.1 三级缩放各自的输入

战略图按缩放级别加载不同粒度的**独立地图包**，每包含风格化底图 + 边界索引图 + 矢量数据：

| 缩放级别 | 地图包 | 包内容 | 用途 |
|---------|--------|--------|------|
| **L3 大世界** | `continent.tres` + 纹理 | 风格化底图(4096-8192) + 边界索引图 + 河流矢量 + 矢量多边形 | 看全大陆，预渲染的草坪/火山/山脉 + 政治边界 |
| **L2 地区** | `region_<id>.tres` + 纹理 | 风格化底图(1024-2048) + 边界索引图 + 地标列表 + 矢量多边形 | 看一个地区，预渲染底图 + 动态地标图标 |
| **L1 地块** | `tile_<id>.tres` + 纹理 | 风格化底图(512-1024) + 边界索引图 + 聚落数据 + 道路矢量 + 资源点 | 看一个地块，底图 + **动态聚落建筑群** + 道路 + 资源 |

> **关键**：每个层级的地图包是**独立的**，粒度切换 = 卸载旧包 + 加载新包。不是同一张图的不同着色逻辑。详见 [`战略图架构.md`](战略图架构.md) §4.2。

### 3.2 region_mask 索引图规范（P社 provinces.bmp 同款）

战略图模块沿用 P社 Clausewitz 引擎的省份索引图机制（[`战略图架构.md`](战略图架构.md) §三详解）：

- **格式**：PNG RGB，每像素的 R/G/B 三通道编码一个 `region_id`
- **编码**：`region_id = (R << 16) | (G << 8) | B`，最大支持 1677 万个地块
- **ID 分配规则**：
  - L3 级别：每个地区一个 `region_id`，从 1 开始连续编号
  - L2 级别：每个地块一个 `region_id`，**与 L3 的地区 ID 独立命名空间**（避免冲突，前缀区分）
  - L1 级别：每个聚落一个 `region_id`，**仅在该地块内有效**（聚落 ID 在地块间可重复，配合 tile_id 唯一确定）
- **背景色**：`RGB(0,0,0)` = 海洋/无地块，渲染时跳过

### 3.3 WorldMapData 结构（运行时数据容器）

战略图模块的运行时数据容器（替代现有的 `WorldMapData` 单一粒度设计），按三级粒度分层：

```gdscript
# 详见 战略图架构.md §四
class_name StrategicMapData extends Resource

# L3 大世界
var continent: ContinentData
# L2 地区（按 region_id 索引，懒加载）
var regions: Dictionary  # {region_id: RegionData}
# L1 地块（按 tile_id 索引，懒加载）
var tiles: Dictionary  # {tile_id: TileData}
# 当前缩放级别
var current_granularity: int  # 0=L3, 1=L2, 2=L1
# 当前聚焦的父级 ID（L2 时是 region_id，L1 时是 tile_id）
var focused_parent_id: String
```

## 五、导入器接口设计

### 5.1 导入器定位

导入器是**开发期 Python 工具**，不是运行时 GDScript。它把生成端的 JSON + PNG + numpy 高度场 + 素材库转成 Godot 可直接 `load()` 的 `.tres` 资源 + 风格化底图 PNG + 边界索引图 PNG。一次性运行，产物随包发布。

**位置**：`tools/worldgen/stylize_renderer.py`（风格化底图生成）+ `tools/worldgen/import_to_godot.py`（打包成 Godot 资源）

**与生成器的关系**：导入器在生成器之后运行，输入是生成器的产物 + 素材库。

### 5.2 导入器 CLI

```
# 步骤 1：风格化底图生成（从原始数据 + 素材库合成风格化地图 PNG）
python tools/worldgen/stylize_renderer.py \
    --input-dir tools/worldgen/output/locked \
    --assets-dir stick-world/assets/strategic_kit \
    --output-dir tools/worldgen/output/stylized

# 步骤 2：打包成 Godot 资源（.tres + PNG 到 config/ 和 assets/）
python tools/worldgen/import_to_godot.py \
    --input-dir tools/worldgen/output/locked \
    --stylized-dir tools/worldgen/output/stylized \
    --output-dir stick-world/config/strategic_map \
    --assets-dir stick-world/assets/strategic_map
```

### 5.3 风格化底图生成（stylize_renderer.py）

这是新增的关键工具，负责把"原始数据（掩码/高度场/群系/河流）"合成为"玩家看到的风格化地图底图"。

**输入**：
- 8192×8192 大陆掩码 + 高度场 + 群系图 + 河流矢量
- `assets/strategic_kit/` 素材库（AI 生成 + 人工筛选，见程序化文档 §十）：
  - `terrain/` 按群系分目录的地面纹理（草坪/沙地/雪原/岩石等，无缝拼接）
  - `decorations/` 装饰物贴图（带占地/通行障碍元数据）
  - `landmarks/` Q版地标图标（矿坑/魔力泉/火山/港口/要塞）
  - `mountains/` 山脉符号（沿等高线放置）
  - `rivers/` 河流样式（笔触/宽度/颜色）

**输出**：三个层级的风格化底图 PNG

```
stylized/
├── l3/
│   └── continent_style_base.png        # 4096×4096 或 8192×8192，整大陆风格化
├── l2/
│   ├── region_001_style_base.png       # 1024×1024 ~ 2048×2048，每地区一份
│   ├── region_002_style_base.png
│   └── ...
├── l1/
│   ├── tile_001_style_base.png         # 512×512 ~ 1024×1024，每地块一份（仅地形纹理）
│   ├── tile_002_style_base.png
│   └── ...
└── boundary_masks/
    ├── l3_continent_boundary.png       # L3 边界索引图（每地区一色）
    ├── l2_region_001_boundary.png      # L2 边界索引图（每地块一色）
    ├── l1_tile_001_boundary.png        # L1 边界索引图（聚落领地）
    └── ...
```

### 5.4 风格化底图合成算法

#### L3 大世界底图合成

```
1. 从 8192 群系图降采样到目标尺寸（4096 或 8192）
2. 按群系铺设地面纹理：
   · 平原群系 → 草坪纹理 repeat + 微扰动（fBm 域扭曲避免机械重复）
   · 荒漠群系 → 沙地纹理 repeat
   · 冰原群系 → 雪原纹理 repeat
   · 森林群系 → 草坪底 + 森林装饰物散布
3. 按高度场放置山脉符号：
   · 山脊线（高度梯度大处）沿走向放置山脉符号
   · 山脉符号大小随高度递增
4. 按河流矢量绘制河流：
   · 河流笔触 + 宽度（按流量）
   · 弯曲处加细节
5. 火山群系特殊处理：
   · 放置火山图标（从 landmarks/volcano/ 选）
   · 周围加岩石装饰
6. 海洋区域：
   · 深浅渐变（按离岸距离）
   · 海岸线描边
7. 输出 continent_style_base.png
```

#### L2 地区底图合成

```
1. 从 8192 原始数据裁切该地区的地理范围
2. 同 L3 的群系铺设 + 山脉 + 河流，但用更高分辨率（因为范围小）
3. 不放置地标图标（地标是 L2 运行时动态放置）
4. 不放置聚落建筑群（聚落是 L1 动态生成）
5. 输出 region_<id>_style_base.png
```

#### L1 地块底图合成

```
1. 从 8192 原始数据裁切该地块的地理范围
2. 仅铺设地形纹理（草坪/沙地/雪原的底色和纹理）
3. 绘制河流（从矢量）
4. 不放置聚落、道路、资源（这些全部运行时生成）
5. 输出 tile_<id>_style_base.png（只是"地形画布"）
```

> **关键差异**：L3 底图是"画好的完整地图"（含草坪/火山/山脉/河流），L1 底图只是"地形画布"（聚落/道路/资源全部运行时生成）。因为 L1 的聚落规模会随玩家建设实时变化，不能预渲染。

### 5.5 边界索引图生成

边界索引图（用于运行时 Shader 着色和点击查询）用 **NEAREST 采样**保持边界锐利：

```
L3 边界索引图：
  · 从 continent_data.json 的地区多边形栅格化
  · 每地区一个唯一 RGB（region_id 编码）
  · 海洋 = RGB(0,0,0)
  · 尺寸同 L3 底图

L2 边界索引图：
  · 从 regions_data.json 的地块多边形栅格化
  · 每地块一个唯一 RGB（tile_id 编码）
  · 尺寸同 L2 底图

L1 边界索引图：
  · 从 tiles_data.json 的聚落领地多边形栅格化
  · 每聚落一个唯一 RGB（settlement_id 编码）
  · 无聚落区域 = RGB(0,0,0)
  · 尺寸同 L1 底图
```

### 5.6 分辨率策略

每个层级按"全屏观看时不出现像素块"为标准，按实际地理范围选合适分辨率：

| 层级 | 底图分辨率 | 边界索引图分辨率 | 理由 |
|------|-----------|---------------|------|
| L3 大世界 | 4096×4096 或 8192×8192 | 同底图 | 整大陆要支持缩放看细节 |
| L2 地区 | 1024×1024 ~ 2048×2048（按地区大小） | 同底图 | 单地区范围小，1024 够清晰 |
| L1 地块 | 512×512 ~ 1024×1024（按地块大小） | 同底图 | 单地块更小，512 够清晰 |

> **不降采样到 1024 一刀切**：每层级按实际地理范围选合适分辨率。L3 最大（整大陆），L1 最小（单地块），各自的全屏观看都清晰。

### 5.7 打包成 Godot 资源（import_to_godot.py）

```
1. 读取 continent_data.json + 风格化底图 + 边界索引图
   ├─ 生成 config/strategic_map/continent.tres（ContinentData）
   │   · 含 style_base_texture 引用
   │   · 含 boundary_mask_texture 引用
   │   · 含 world_bounds、regions 列表
   └─ 复制 PNG 到 assets/strategic_map/l3/

2. 读取 regions_data.json
   对每个地区 region_id：
   ├─ 生成 config/strategic_map/regions/region_<id>.tres（RegionData）
   │   · 含 style_base_texture、boundary_mask_texture、world_bounds、neighbors
   └─ 复制 PNG 到 assets/strategic_map/l2/

3. 读取 tiles_data.json
   对每个地块 tile_id：
   ├─ 生成 config/strategic_map/tiles/tile_<id>.tres（TileData）
   │   · 含 style_base_texture、boundary_mask_texture、world_bounds、neighbors
   │   · 含 settlements、resources、rivers、roads
   └─ 复制 PNG 到 assets/strategic_map/l1/

4. 读取 political_data.json
   └─ 生成 config/strategic_map/political.tres（PoliticalData）

5. 生成 manifest.tres（索引清单，列出所有 .tres 和 .png 路径）
```

### 5.8 拼接预览的边缘对齐约定

地图包之间的边缘必须对齐，拼接时无缝（见 [`战略图架构.md`](战略图架构.md) §5）：

- 每个地图包记录其在世界坐标系中的**地理范围**（`world_bounds: Rect2`，0-8192 坐标系）
- 拼接时按 `world_bounds` 计算偏移，相邻包的边缘像素天然连续（因为都源自同一张 8192 原始数据裁剪）
- 边界索引图在包边缘有 1-2 像素的重叠区（`edge_overlap`），避免拼接缝隙
- 邻接关系（`neighbors`）从程序化生成的邻接图直接导入

## 六、数据结构映射：TileData ↔ RegionDefinition

这是 [`程序化世界生成.md`](../../设计/系统/08-程序化世界生成.md) §7.2 的 `TileData` 与现有 `world_map/data/region_definitions.gd` 的 `RegionDefinition` 的字段映射。

### 6.1 现有 RegionDefinition 的问题

现有 `RegionDefinition` 是单一粒度的「地块」定义，混入了：
- **L1 地块级属性**（resource_types / stickman_types / tech_unlocks）
- **L3/L2 政治属性**（initial_owner / adjacent_region_ids）
- **渲染属性**（center_position / outline_points）

新架构按三级粒度拆分，`RegionDefinition` 拆成三个类。

### 6.2 新数据模型（详见 战略图架构.md §四）

#### ContinentData（L3 大世界）

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `continent_id` | String | 固定 `"continent_main"` | 唯一大陆 |
| `seed` | int | `locked_continent` 的 seed | 锁定种子，用于按需生成 |
| `size` | Vector2i | 8192×8192 | 原始数据尺寸 |
| `regions` | Array[String] | `continent_data.json` 的 region_id 列表 | 该大陆所有地区 ID |
| `ocean_mask` | Image | `locked_continent_8192.png` | 海洋掩码（用于河流裁剪、海岸线绘制） |

#### RegionData（L2 地区）

| 字段 | 类型 | 来源 | 对应程序化文档字段 |
|------|------|------|------------------|
| `region_id` | String | `"region_001"` | §5.5 地区种子 |
| `name` | String | `continent_data.json` | 地区名 |
| `polygon` | PackedVector2Array | `continent_data.json` | §5.5 地区多边形 |
| `adjacent_regions` | Array[String] | `continent_data.json` | §5.5 邻接图 |
| `dominant_biome` | int | `regions_data.json` | §0.4 主导气候 |
| `tiles` | Array[String] | `regions_data.json` | 该地区所有地块 ID |
| `political_owner` | String | `political_data.json` | 当前归属政权（可运行时改） |
| `alliance` | String | `political_data.json` | 所属联盟（可空） |

#### TileData（L1 地块）

| 字段 | 类型 | 来源 | 对应程序化文档字段 |
|------|------|------|------------------|
| `tile_id` | String | `"tile_001"` | §7.2 tile_id |
| `parent_region_id` | String | `regions_data.json` | 所属地区 |
| `polygon` | PackedVector2Array | `regions_data.json` | §7.2 polygon |
| `biome` | int | `regions_data.json` | §7.2 biome |
| `settlements` | Array[SettlementRef] | `tiles_data.json` | §7.2 settlements |
| `resources` | Array[ResourceDeposit] | `tiles_data.json` | §7.2 resources |
| `rivers` | Array[PackedVector2Array] | `tiles_data.json` | §7.2 rivers（裁剪自 L3） |
| `central_city_id` | String | `tiles_data.json` | §7.2 central_city_id |
| `harshness` | float | `tiles_data.json` | §3.2 harshness（影响聚落密度） |

#### SettlementRef（聚落引用，轻量）

```gdscript
class_name SettlementRef extends Resource
var settlement_id: String       # "settlement_tile_001_03"
var name: String
var level: int                  # 1=村落 T1, 2=镇 T2, 3=城市 T3, 4=中心城市 T4, 5=首都 T5
var industry: Array[String]     # 支柱产业标签 ["mining", "trade"]
var position: Vector2           # 在地块多边形内的归一化坐标（0-1）
var population_score: float     # 规模分数（运行时可被 ±15% 扰动）
var map_id: String              # 对应场景图的 map_id（点击进入时 load_map 用）
```

> **关键**：`SettlementRef` 只存战略图需要的展示信息，不存场景图内部的建筑/装饰数据。点击聚落进入场景图后，由 `SettlementGenerator` 从 `settlement_seed` 按需生成 `MapDefinition.tres`。

### 6.3 旧 RegionDefinition 的迁移

现有 `RegionDefinition` 字段在新模型中的去向：

| 旧字段 | 新去向 | 说明 |
|--------|--------|------|
| `id: int` | `tile_id: String`（TileData） | 类型改 String |
| `name` | `SettlementRef.name`（中心城市）或 `RegionData.name`（地区名） | 按粒度拆分 |
| `type: int`（陆地/海洋/湖泊/荒原） | `biome: int`（TileData） | 改用群系枚举 |
| `is_coastal` | `TileData` 派生属性 | 从 polygon 与海洋掩码计算 |
| `resource_types` | `TileData.resources` | 改为 ResourceDeposit 数组（含量级） |
| `stickman_types` | 删除 | 不属于战略图关注，移到场景图侧的 `VillageDefinition` |
| `tech_unlocks` | 删除 | 不属于战略图关注，移到科技模块的 `tech_unlocks_by_region` |
| `initial_owner` | `RegionData.political_owner` | 上移到地区级 |
| `adjacent_region_ids` | `RegionData.adjacent_regions` | 上移到地区级 |
| `center_position` | `SettlementRef.position`（聚落）或 `RegionData.polygon` 质心 | 按粒度拆分 |
| `outline_points` | `RegionData.polygon` / `TileData.polygon` | 直接用多边形 |

## 七、运行时数据流

### 7.1 战略图启动流程

```
1. GameRoot 启动 → SceneLoader 加载初始场景图（玩家所在村落）
2. 玩家按 Tab 打开战略图 → UIRoot 显示 ModalOverlay
3. StrategicMapController._ready()
   ├─ 加载 config/strategic_map/manifest.tres
   ├─ 加载 continent.tres（L3，必加载）
   └─ 默认 granularity = L3，渲染全大陆
4. 玩家滚轮缩放 → 触发 granularity 切换
   ├─ L3 → L2：根据点击的 region_id 加载 region_<id>.tres + region_mask_l2_<id>.png
   ├─ L2 → L1：根据点击的 tile_id 加载 tile_<id>.tres + region_mask_l1_<id>.png
   └─ L1 → 场景图：点击聚落 → 关闭战略图 → SceneLoader.load_map(settlement.map_id)
```

### 7.2 懒加载策略

- L3 大陆地图包**启动时必加载**（1 份，~10-50MB 含纹理）
- L2 地区地图包**双击地区时懒加载**（每地区 ~2-5MB，全大陆约十来个地区）
- L1 地块地图包**双击地块时懒加载**（每地块 ~0.5-2MB，全大陆数十到数百个地块）
- 同一时刻最多加载：1 个 L3 + 1 个 L2 + 1 个 L1（拼接预览模式除外，见 [`战略图架构.md`](战略图架构.md) §5）
- 拼接预览模式：1 个中心包 + 4-8 个相邻包（降采样）+ 远邻矢量描边

### 7.3 与场景图的切换契约

战略图 ↔ 场景图的切换通过 `SceneLoader` + EventBus 信号：

| 事件 | 触发 | 数据流 |
|------|------|--------|
| 玩家在场景图按 Tab | `InputDispatcher` | → 打开战略图 ModalOverlay，记录 `return_map_id` |
| 玩家在战略图双击聚落 | `StrategicMapController` | → `EventBus.travel_requested(settlement.map_id)` → `SceneLoader.load_map()` |
| 玩家在战略图按 ESC | `StrategicMapController` | → 关闭 ModalOverlay，回到 `return_map_id` 场景图 |
| 场景图内聚落规模变化 | construction 模块 | → `EventBus.settlement_updated(settlement_id)` → L1 地图包重新渲染该聚落建筑群 |
| 场景图内归属变化 | expansion 模块 | → `EventBus.region_owner_changed(region_id)` → 战略图刷新政治着色 |
