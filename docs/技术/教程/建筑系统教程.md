## 一、行业最佳实践

### 1.1 核心原则：工作流分层，永不互盖

业内所有成功案例（THE FINALS、Fortnite、Townscaper、Houdini）有一个共同点：

**代码生成和手动编辑互不覆盖。**

```
模板层（代码）  →  定义建筑的结构约束（宽度、材质、柱间距……）
生成层（代码）  →  据模板一次性生成节点树，写入 .tscn，之后永不动
编辑层（美术）  →  在 Godot 编辑器里调整节点位置/角度/纹理，Ctrl+S 持久化
```

| 项目 | 方法 | 关键洞察 |
|------|------|---------|
| THE FINALS (Embark) | Blockout → Feature Nodes | 美术摆粗模 → 程序自动化填充细节 |
| Fortnite / AAA 通用 | Modular Grid Kit | 标准模块（墙/柱/顶）像乐高，定义贴附规则后批量实例化 |
| Townscaper (Stålberg) | Wave Function Collapse | 纯约束驱动，无预制场景，用户放一块系统推测周围 |
| Houdini PDG | 节点图参数化 | 一套参数化网络 → 无数建筑变体，每次重建整个节点树 |

### 1.2 本项目的独特性

标准 AAA 流程是"美术先做模块 → 程序批量摆"。但本项目是**反过来的**：

> 先手工搭一个"完美参考建筑" → 反推参数 → 生成系统复现 → 扩展到不同尺寸/配置。

原因：一个人开发，没有专业像素美术，必须先看到"我要做的东西长什么样"才能定义约束规则。

### 1.3 为什么不能"代码和编辑器同时改"

```
场景 tscn = 代码写入 + 用户手动编辑

代码写入 tscn ──┐
                ├──→ 后写者覆盖前者 → 手工调整丢失
用户手动编辑 ───┘
```

Godot 编辑器和代码都对同一个 `.tscn` 文件进行写操作，两者没有合并逻辑。
**解决**：代码只负责"首次填充空场景"，之后退出。场景变纯编辑器资产。

## 二、参考建筑 → 参数化 → 生成  完整流程

### 2.1 第一步：手搭参考场景

在 Godot 编辑器中直接新建一个 `Node2D` 场景，摆好所有子节点：

```
SmithyReference (Node2D)
├── L1_BackWall
│   ├── BackWall (Sprite2D, 222×102 thatch)
│   ├── BackWallTop (Sprite2D, smaller upper section)
│   ├── BackPillarL (Sprite2D, wood pillar)
│   └── BackPillarR (Sprite2D, wood pillar)
├── L2_BackItems (空容器，后期放后排物品)
├── L3_FrontItems (空容器，后期放前排物品)
├── L4_FrontWall
│   ├── FrontPillar ×3 (Sprite2D, wood pillar)
└── L5_Roof
    ├── SlantedBeam (Sprite2D, rotated wood)
    ├── VerticalStrut (Sprite2D, wood)
    ├── Beam (Sprite2D, 238×16 wood beam)
    ├── MainRoofGroup (容器)
    │   ├── RoofRightEnd (Polygon2D, 平行四边形草皮)
    │   └── RoofMain (Polygon2D, 平行四边形草皮)
    ├── SlantedStrut (Sprite2D, rotated wood)
    └── RoofLeftGroup1 (Polygon2D, 平行四边形草皮)
```

**命名规则**：所有节点用英文，层容器用 `L1_` ~ `L5_` 前缀。
**渲染顺序**：Godot 按节点树顺序（后添加的在上面）渲染，不依赖 `z_index`。

### 2.2 第二步：提取坐标参数

从 `.tscn` 文件读取每个节点的精确坐标，写为脚本常量：

```gdscript
# --- 后景墙壁 ---
const BW_MAIN_POS      = Vector2(83, -250.5)
const BW_MAIN_SCALE    = Vector2(1.027, 0.755)
const BW_TOP_POS       = Vector2(36.375, -312.375)

# --- 屋顶：主屋顶（平行四边形拆解为 4 参数） ---
const RM_BL            = Vector2(103, -236)   # 左下角
const RM_DW            = 91.0                # 底边宽度
const RM_TY            = -346.0              # 顶边 Y
const RM_SLANT         = -64.0               # 斜边偏移（负=向左斜）
```

**平行四边形参数化公式**：

- 左下角 = `bl`
- 右下角 = `(bl.x + dw, bl.y)`
- 右上角 = `(bl.x + dw + slant, top_y)`
- 左上角 = `(bl.x + slant, top_y)`

**60° 斜边的验证**：`|slant| ≈ (bl.y - top_y) / √3`（即高度 × cot(60°) ≈ 高度 × 0.577）

### 2.3 第三步：编写 @tool 构建器

```gdscript
@tool
extends Node2D

var _done = false

func _ready():
    if Engine.is_editor_hint():
        if has_node("L1_BackWall"):   # 已有节点 → 跳过
            return
        if not _done:
            _done = true
            call_deferred("_build")
```

**关键设计**：
- `has_node("L1_BackWall")` → 检测场景是否已生成过。生成过的就是纯编辑器资产，**代码不再碰**。
- `call_deferred("_build")` → 延迟一帧，让编辑器完全初始化。
- `owner = self` 必须在 `add_child` 之后设置（Godot 要求 owner 是树中的祖先）。

### 2.4 第四步：验证 + 保存

1. 打开 `smithy_preview.tscn` → 自动生成节点
2. **Ctrl+S 保存** → 节点写入 tscn，纹理也嵌入
3. 关闭重开 → 检测到 `L1_BackWall` 存在 → 跳过生成
4. 现在任意调整节点位置/旋转/纹理，Ctrl+S 持久化
5. 把调整后的坐标反哺回脚本（如果参考试验满意）

## 三、五种材质建筑模板（待实现）

本节结合游戏设计文档中 4 种基础材质 + 1 种超大建筑的业务需求。

| 材质 | 层数 | 墙体 | 柱体 | 屋顶 | 典型宽度 |
|------|------|------|------|------|---------|
| 稻草棚 (thatch) | 1 | 无墙 | 前后柱 | 人字草顶 | 3-16 格 |
| 木板房 (wood) | 2 | 前后木板 | 无 | 人字木板顶 | 3-8 格 |
| 石头房 (stone) | 3 | 前后石墙 | 角石 | 平顶/人字 | 3-8 格 |
| 砖瓦房 (brick) | 4 | 前后砖墙 | 砖垛 | 人字/平顶 | 3-12 格 |
| 大型宫殿/地下城 | 不限 | 混合 | 混合 | 自定义 | 不限 |

> 大型建筑（宫殿、地下城）保留预制场景方式（`old_buildings/`），不走程序化生成。

### 3.1 渲染系统（三层协同）

建筑在场景图（卷轴地图）中渲染，5 层从后到前：

```
L1 后景墙壁（+ 后景屋顶作为其一部分）
L2 后排物品
L3 前排物品
L4 前景墙壁
L5 屋顶（落在 L1-L4 之上）
```

**Y-sort**：整个建筑作为一个整体参与场景图的 Y-sort，不在建筑内部做 Y-sort。

### 3.2 屋顶 60° 斜边规则（硬约束）

#### 数学约束

所有 thatch 屋顶平行四边形斜边与地面成 **60° 夹角**，不可偏离：

```
slant = height / tan(60°) = height / 1.73205
```

验证参考建筑三个屋顶片：

| 屋顶片 | 高度 (px) | slant | slant/height | 与 1/1.732 偏差 |
|--------|----------|-------|-------------|-----------------|
| RoofLeftGroup1 | 179 | 103 | 0.575 | -0.4% |
| RoofMain | 110 | 64 | 0.582 | +0.8% |
| RoofRightEnd | 140 | 81 | 0.579 | +0.2% |

> 偏差来自像素取整，可接受范围内。

#### 拉伸规则（关键！）

建筑变宽时：

| 元素 | 行为 | 说明 |
|------|------|------|
| **RoofMain** | **只拉伸上下两边（dw 增加）** | 斜边角度固定 60°，左右倾斜边只平移不旋转 |
| **RoofLeftGroup1** | 完全固定 | 左侧端盖，包含悬挑部分 |
| **RoofRightEnd** | 位置跟随 RM 右端 | 形状固定（36px 底边），tl 顶角贴合 RM.tr |
| **SlantedStrut** | 完全固定 | 斜支柱右上角与 RL1 右上角重合 |
| **BackWall** | 宽度拉伸 | 后景墙壁纹理随宽度缩放 |
| **Beam** | 长度拉伸 | 横梁跨度随宽度增加 |
| **前景柱** | 数量 + 间距 | `max(3, ceil(width_cells/4))` 根，均匀分布 |

#### 为什么只拉伸 RoofMain

如果拉伸整个屋顶（含端盖），60° 斜边会被拉伸变形，不再是 60°。正确做法：

- **RoofMain 拉伸**：上下两边延长 → 斜边平移，角度不变
- **RoofLeftGroup1 / RoofRightEnd 固定**：保持原尺寸，贴合 RM 两端
- **RoofRightEnd 自动跟随**：其 tl 顶点始终等于 RM.tr 的世界坐标

```
参考 (7格):  RL1 |<- RM(dw=87.17) ->| RR
18格:       RL1 |<- RM(dw=224.15) --------->| RR
                  ↑ 只这一段变长
```

#### SlantedStrut 几何

支柱为 144×20 centered Sprite2D，旋转 60°（1.047 rad）。左上角锚定 RL1 右上角 (53, -358)，下延穿透 Beam(y≈-229)：

```
左上偏移 = rotate(-72, -10) @60° = (-27.34, -67.35)
中心 = (53 + 27.34, -358 + 67.35) = (80.34, -290.65)
右下角 = (107.68, -223.30)   ← 触及 Beam 内
```

## 四、程序化纹理系统

### 4.1 设计决策

本项目的所有建筑纹理使用**程序化生成**（GDScript 代码画像素），不使用外部 PNG。

原因：
- 一个人开发，没有像素美术
- 程序化纹理可随意调参数（颜色、噪点密度、条纹间距）
- 同一份代码生成木纹/稻草/石头/砖块多种纹理
- 纹理尺寸可变，匹配不同建筑宽度

### 4.2 当前可用纹理

```
procedural_materials.gd:
├── make_wood_pillar(w, h, color)      → 竖条纹木纹理
├── make_wood_pillar_horizontal(w, h)  → 横条纹木纹理
└── make_straw_thatch(w, h, color)     → 稻草纤维纹理（随机线段 + 噪点）
```

### 4.3 未来扩展

- `make_stone_wall(w, h)` → 随机石块拼接
- `make_brick_pattern(w, h)` → 规则砖块
- `make_wood_planks(w, h)` → 木板拼接

## 五、模块与物品系统（未来）

### 5.1 模块的概念

"模块"是建筑内部的独立功能单元。玩家盖建筑时像搭积木 —— 选尺寸 → 选材质 → 放模块（锻造台/床/货架）。

模块用 `interior_scene.tscn` 定义：
- 含装饰 Sprite（如铁砧、桌子）
- 含工作位 Marker2D（火柴人站在这里工作）
- 含占位信息（几格宽、是否需要前景开口）

模块与建筑模板是正交概念：
- 铁匠铺 = 稻草棚建筑 + 锻造作坊模块
- 民居 = 木板房建筑 + 住所模块
- 高级铁匠铺 = 砖瓦房建筑 + 锻造作坊模块

## 六、避坑指南（Day 1 血泪总结）

### 6.1 千万不要反复写 tscn

**症状**：你在编辑器里调了半天位置/角度 → AI 重写 tscn → 全丢了。

**根治**：`_ready` 里 `if has_node("L1_BackWall"): return`。场景一旦有了节点，代码就跑，再跑就覆盖。做到"首次填充后退出"。

### 6.2 不要用重建按钮

**症状**：`@export var rebuild: bool = false` → 你勾一下就擦掉所有手调。

**根治**：删除 rebuild 机制。场景生成了一次就永远是它自己的样子。想重新生成 = 删场景重新打开。

### 6.3 Grass（草坪底盘）的教训

**症状**：用户反复说"草坪删掉"，但因为脚本每打开就重建 → 删不掉。

**根治**：同上。"首次填充后退出"模式下，用户删完 Ctrl+S，下次打开不存在就是不存在。

### 6.4 标记工具做复杂了

**症状**：想做个标注工具让用户打标记 → 搞了独立场景 → 用户不知道怎么挂 → 标注失效。

**更好的办法**：直接读用户调好的 `.tscn` 文件，用脚本提取坐标。不需要任何额外工具。

### 6.5 match 多臂不能写在同一行

```gdscript
# 错误（Godot 4.x 不允许）
match event.keycode:
    KEY_LEFT: m.x -= 50; KEY_RIGHT: m.x += 50

# 正确
match event.keycode:
    KEY_LEFT:  cam.position.x -= 50
    KEY_RIGHT: cam.position.x += 50
```

### 6.6 tscn 嵌入纹理导致巨大文件

**症状**：966KB 的 tscn（纹理嵌入）。

**解决**：纹理用代码生成（`procedural_materials.gd`），不嵌入 tscn。纹理在 `_ready` 中创建为 `ImageTexture`，不持久化到 tscn。

### 6.7 owner 必须设置时节点已在树中

```gdscript
# 错误
var s = Sprite2D.new()
s.owner = self    # 报错：owner 必须是树中的祖先
add_child(s)

# 正确
var s = Sprite2D.new()
add_child(s)
s.owner = self    # 此时 self 在树中
```

### 6.8 @tool 脚本引用 Autoload 类时的陷阱

**症状**：`Invalid call function 'register_roof' in base 'Node (BuildingGenApi)': Attempt to call a method on a placeholder instance`

**原因**：`BuildingGenApi` 的脚本（`api.gd`）缺少 `@tool` 标记。Godot 编辑器对非 `@tool` 的 Autoload 只加载为**占位实例**（placeholder），不执行其完整 `_ready()` / 方法体。

**根治**：所有需要在编辑器预览中被 `@tool` 脚本引用的 Autoload 类，必须在脚本顶部加 `@tool`：

```gdscript
@tool
class_name BuildingGenApi
extends Node
```

同样适用于所有可能在编辑器预览中实例化的 **Resource 子类**（`BuildingDef`、`MaterialDef` 等）。不加 `@tool` 的 Resource 类在编辑器模式下只有字段没有方法。

### 6.9 文件路径引用：模块根目录 vs scripts/ 子目录

**症状**：`Attempt to open script 'res://modules/building_gen/scripts/api.gd' resulted in error 'File not found'`

**原因**：按模块化架构规范，`api.gd` 放在模块根目录（`modules/building_gen/api.gd`），而非 `scripts/` 子目录。

**根治**：区分两类文件路径：

| 文件类型 | 位置 | 示例 |
|---------|------|------|
| 公共 API | 模块根目录 | `res://modules/building_gen/api.gd` |
| 内部脚本 | `scripts/` 子目录 | `res://modules/building_gen/scripts/preview/smithy_preview.gd` |
| 场景文件 | `scenes/` 子目录 | `res://modules/building_gen/scenes/smithy_preview.tscn` |

### 6.10 Polygon2D 平行四边形屋顶的 UV 映射

**问题**：稻草棚屋顶是平行四边形草皮，不能直接用矩形 Sprite，需要 `Polygon2D` + 自定义 UV。

**公式**（4 顶点 → 4 UV 坐标）：

```gdscript
func _para(pname: String, bl: Vector2, dw: float, ty: float, sl: float, tex) -> Polygon2D:
    # bl = 左下角, dw = 底边宽, ty = 顶边Y, sl = 斜边X偏移
    var br = Vector2(bl.x + dw, bl.y)         # 右下角
    var tr = Vector2(bl.x + dw + sl, ty)       # 右上角
    var tl = Vector2(bl.x + sl, ty)            # 左上角
    var p = Polygon2D.new()
    p.polygon = PackedVector2Array([tl, tr, br, bl])  # 逆时针
    # UV：按底边宽度等比缩放，让纹理水平拉伸匹配平行四边形宽度
    var us = bl.distance_to(br) / float(tex.get_image().get_width())
    p.uv = PackedVector2Array([Vector2.ZERO, Vector2(us, 0), Vector2(us, 1), Vector2(0, 1)])
    p.texture = tex
    return p
```

**关键点**：
- 多边形顶点**逆时针**排列（Godot 要求）
- UV 的 U 分量按底边宽度 / 纹理宽度缩放，V 分量 0→1 不变
- 这种映射让 64×64 的纹理在任意宽度的平行四边形上正确显示，不需要为每个宽度创建专用纹理

### 7.2 参数化构建核心机制

**统一 shift 平移**：建筑加宽时，只拉伸 RoofMain(dw)、BackWall(宽度)、Beam(长度)。右侧所有元素（RR、BackPillarR、最右前柱）平移相同距离：

```
shift = RM.tr(world).x - 128.91    # RM 右端世界坐标偏移量
```

| 元素 | 宽度变化 |
|------|---------|
| RoofMain | dw = 87.17 × s |
| BackWall | half_w = 114 + shift/2 |
| Beam | half_w = 146 + shift/2 |
| BackPillarR | x = 166 + shift |
| 最右前柱 | x = 204 + shift |

**堵头分区**：
- **左堵头**（固定）：RL1、BackPillarL、VerticalStrut、SlantedStrut、SlantedBeam、BackWallTop、前柱第1-2根
- **右堵头**（平移）：RR、BackPillarR、最右前柱
- **中段拉伸**：RoofMain、BackWall、Beam + 额外柱

**柱子密度**：`ceil(width_cells / 6) + 1`，前后柱同步。
