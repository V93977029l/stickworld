# 建筑系统教程

> 本教程介绍如何添加建筑素材、配置建筑数据、在游戏内放置建筑。
> 关联架构：`docs/技术/架构/场景与战斗架构.md` §4 建筑系统、`docs/技术/架构/存储架构设计.md`

---

## 一、建筑系统总览

### 数据流

```
建筑数据表 (xlsx) -> buildings.tres (静态模板) ← 只读
                                        ↓
游戏内放置 -> PlacementGrid 占用 -> 存档 JSON (动态实例) ← 可写
```

### 三层结构

| 层 | 说明 | 格式 | 可写性 |
|----|------|------|--------|
| 建筑模板 | 建筑属性（HP/成本/科技依赖） | `.tres`（从 xlsx 生成） | 只读 |
| 建筑素材 | 美术资源（精灵/动画） | `.png`/`.tscn` | 只读 |
| 建筑实例 | 游戏内放置的实例 | 存档 JSON | 可写 |

**关键原则**：游戏内建造**不修改** `.tres`，只写入存档 JSON。详见 `docs/技术/架构/存储架构设计.md`。

## 二、添加建筑素材

### 步骤 1：准备美术资源

建筑素材规范：
- **格式**：PNG（透明背景）
- **尺寸**：**任意**，贴图尺寸与占地尺寸解耦（详见 §二贴图与占地解耦）
  - 占地宽度 = 条带数 × 32px（决定 PlacementGrid 占用）
  - 贴图尺寸可以比占地宽或窄
  - 例：一棵树占地 1 格宽（32px 树根），贴图可 96×192px（树冠很宽很高）
- **朝向**：默认正面朝左下（火柴人视角）
- **风格**：与火柴人低多边形/简笔风格一致

### 步骤 2：放置素材文件

将 PNG 放到：
```
modules/buildings/assets/textures/
├── house_wood.png
├── farm.png
└── workshop.png
```

### 步骤 3：创建建筑独立场景（必需）

**所有建筑统一使用独立场景文件**，即使是最简单的纯 Sprite2D 建筑。这是为了：
1. 预置建筑（地图设计时放置）和运行时建筑（玩家建造）走同一套加载路径
2. 建筑复杂度升级（加动画/子节点）时不需要改加载逻辑
3. 与建筑 ID 一一对应，方便 `BuildingRegistry` 索引

**场景路径约定**（与建筑 ID 严格对应）：
```
modules/buildings/scenes/<building_id>.tscn
```

例如：
```
modules/buildings/scenes/
├── bld_house.tscn       (id=bld_house)
├── bld_farm.tscn        (id=bld_farm)
├── bld_workshop.tscn    (id=bld_workshop)
├── bld_barracks.tscn    (id=bld_barracks)
├── bld_market.tscn      (id=bld_market)
└── bld_academy.tscn     (id=bld_academy)
```

**场景结构规范**：

```
BuildingRoot (Node2D, 根节点)
├── Sprite2D       (建筑贴图，offset 让底部对齐原点)
├── PassageBarrier (Area2D, 建筑本体通行障碍)
│   └── CollisionShape2D[] (RectangleShape2D, 矩形障碍区域)
├── AnimationPlayer (可选，建造/受损动画)
└── HealthBar       (可选，血条)
```

**关键约定**：
- 根节点必须是 `Node2D`，命名 `BuildingRoot`（或与 ID 一致）
- 建筑底部（脚底）必须对齐根节点原点 `(0, 0)`
- 即 Sprite2D 的 `offset.y = -texture_height / 2`
- 这样放置时 `position = (cell_x * 32, ground_y)` 即可，无需额外偏移
- **PassageBarrier 是必需节点**（即使为空也要存在），用于标记建筑本体不可通行区域

**简单建筑示例**（`bld_house.tscn`）：

```gdscript
# 直接在 Godot 编辑器中创建：
# Node2D (name=BldHouse)
# └── Sprite2D (texture=house_wood.png, offset=(0, -32))
# Sprite2D.offset.y = -32 让 64×64 贴图底部对齐 BuildingRoot 原点
```

**复杂建筑示例**（带血条和动画）：

```
BldBarracks (Node2D)
├── Sprite2D (texture=barracks.png, offset=(0, -48))
├── AnimationPlayer (build_anim, damage_anim)
└── HealthBar (TextureProgressBar, visible=false)
```

### 步骤 4：导入设置

在 Godot 编辑器中选中 PNG，设置 Import：
- **Preset**：2D Pixel（如果是像素风）或 2D
- **Filter**：关闭（像素风）或开启（平滑风）
- **Compression**：Lossy（质量 0.7）

## 二·补、贴图与占地解耦

**核心原则**：建筑贴图尺寸任意，与占地尺寸完全解耦。

| 概念 | 决定什么 | 由谁配置 |
|------|---------|---------|
| **占地宽度**（footprint_w） | PlacementGrid 占用条带数 | xlsx 数据表 |
| **贴图尺寸** | 视觉表现范围 | PNG 文件本身 |

**典型用例**：

| 建筑 | 占地 | 贴图尺寸 | Sprite2D offset | 说明 |
|------|------|---------|-----------------|------|
| 民居 | 2 格宽（64px） | 64×64px | (0, -32) | 贴图刚好填满占地 |
| 大树 | 1 格宽（32px） | 96×192px | (-32, -160) | 树根占地 1 格，树冠向上向左展开 |
| 雕像 | 1 格宽（32px） | 32×64px | (0, -32) | 窄高贴图，底部对齐 |
| 兵营 | 3 格宽（96px） | 128×128px | (-16, -64) | 贴图略大于占地，向上扩展 |

**关键**：
- PlacementGrid 只校验水平条带占用，不管贴图是否超出
- 贴图超出占地范围不冲突，多个建筑贴图可以视觉重叠（如树冠交叉）
- Sprite2D 的 `offset.y = -texture_height / 2` 让底部对齐 BuildingRoot 原点
- 贴图水平方向偏移用 `offset.x`（负值向左，正值向右）

## 二·补、PassageBarrier 建筑通行障碍

PassageBarrier 标记建筑本体哪些区域火柴人不可穿越。与地图级 `WalkBarrier` 共用同一套碰撞查询（详见架构 §7.1.2）。

### 典型用例

| 建筑类型 | 通行规则 | PassageBarrier 配置 |
|---------|---------|---------------------|
| 雕像（小） | 角色从上下两侧都可绕过 | 本体一小块矩形（仅雕像基座） |
| 房屋 | 角色只能从房屋下方通过 | 覆盖房屋上半部分（墙体），下方留空 |
| 兵营 | 角色可从下方过，正门交互进入 | 覆盖建筑上半部分 + 左右两侧 |
| 矿山（地形建筑） | 角色完全不能穿过 | 整个占地范围全覆盖 |
| 树木（装饰） | 角色可绕过 | 仅树根一小块 |

### 设计原则

- PassageBarrier 形状可与占地不同（占地是水平条带，PassageBarrier 是物理碰撞）
- 例：房屋占地 3 格宽（96px），PassageBarrier 只覆盖地面上半部分，下方留给角色通行
- 设计时在建筑场景编辑器直接绘制 CollisionShape2D，所见即所得
- 调试显示：DebugOverlay 开启时，建筑 PassageBarrier 显示为**紫色半透明**（与地图级 WalkBarrier 蓝色区分）

### 配置示例

**房屋（只能从下方过）**：
```
BldHouse (Node2D)
├── Sprite2D (texture=house.png, offset=(0, -48))
├── PassageBarrier (Area2D)
│   └── CollisionShape2D (RectangleShape2D, size=96x64, pos=(0, -32))  # 上半部分
└── ...
```
PassageBarrier 覆盖 y ∈ [-64, 0]（上半部分），下方 y ∈ [0, 32] 留给角色通行。

**雕像（上下可绕过）**：
```
BldStatue (Node2D)
├── Sprite2D (texture=statue.png, offset=(0, -32))
├── PassageBarrier (Area2D)
│   └── CollisionShape2D (RectangleShape2D, size=32x32, pos=(0, -16))  # 仅基座
└── ...
```
PassageBarrier 仅覆盖基座 32×32px，角色从上下两侧都可绕过。

## 四、放置建筑（当前状态 + 待实现）

### 地图编辑器工具（MapEditor 插件）

安装后，在 2D 编辑器中打开地图场景即可使用：

1. 工具栏出现"建筑放置"按钮（仅地图场景显示）
2. 网格竖线自动显示（32px 间距，每 8 格加粗）
3. 点击按钮激活放置模式
4. 数字键 1/2/3/4 切换建筑类型
5. 鼠标移动时，建筑将占用的竖条高亮为白色半透明
6. 左键放置建筑（自动吸附网格，底部对齐草地垂直中线）
7. 右键删除附近建筑
8. ESC 退出

建筑放置位置：
- X = 格子边界（32px 对齐）
- Y = (ground_y + ground_bottom) / 2（草地垂直中线）

### 当前已实现（P0）

**PlacementGrid**：32px 精度占地网格
- `occupy(x, w, id)`：占用格子
- `release(id)`：释放格子
- `is_occupied(x)`：查询占用
- `world_to_cell(pos)`：世界坐标->格子坐标
- `cell_to_world(x)`：格子坐标->世界坐标

**PlacementValidator**：放置校验
- `validate_placement(grid, x, y, w, h)`：校验是否可放置（边界+冲突）

代码示例：
```gdscript
var grid: PlacementGrid = map.placement_grid
var v := PlacementValidator.new()
var result = v.validate_placement(grid, cell_x, 2)
if result.ok:
    grid.occupy(cell_x, 2, "bld_0001")
    # 实例化建筑场景到 BuildingHost
    var building := building_scene.instantiate()
    map.building_host.add_child(building)
    building.global_position = grid.cell_to_world(cell_x)
```

## 六、建筑三层架构（地形 / 初始 / 玩家）

游戏内有三种建筑来源，它们**和平共存**，通过同一套独立场景文件加载，但存储位置和可修改性不同。

### 6.1 三层对比

| 层 | 存储位置 | 可拆除 | 加载时机 | 示例 |
|---|---|---|---|---|
| **地形建筑层** | 地图场景 `.tscn` 中 `TerrainBuildings` 节点 | 否 | 随场景实例化 | 矿山、遗迹、大石头 |
| **初始建筑层** | 地图场景 `.tscn` 中 `InitialBuildingsList` 列表 | 是 | 首次进入地图写入存档 | 关卡设计放的民居 |
| **玩家建筑层** | 存档 `.json` | 是 | 存档加载时实例化 | 玩家新建的农场 |

### 6.2 为什么不会冲突？

三者都使用**同一套建筑场景文件**（`modules/buildings/scenes/<building_id>.tscn`），只是加载时机不同。`PlacementGrid` 是统一占位登记表，三层建筑都向它注册，避免格子被重复占用。

```
地图加载流程：
1. SceneLoader 实例化 map.tscn
   └── TerrainBuildings 下的建筑自动随场景创建（静态，注册到 PlacementGrid，标记 is_terrain=true）
2. VillageMap._ready() 读取 InitialBuildingsList
   └── 首次进入该地图：将列表写入存档 JSON 的 buildings 数组（标记 is_initial=true）
   └── 非首次：跳过（存档已有，玩家可能已拆除部分）
3. SaveManager 读取存档 JSON
   └── 对每个动态建筑实例（初始 + 玩家）：
       ├── 加载对应场景文件
       ├── 实例化后 add_child 到 BuildingHost
       └── 注册到 PlacementGrid
```

**PlacementGrid 的角色**：它是统一占位登记表。三层建筑都向它注册，避免格子被重复占用：
- 地形建筑在 `_ready()` 中注册（一次性，标记 `is_terrain=true`）
- 初始建筑首次进入地图时写入存档，之后随存档加载注册
- 玩家建筑在玩家点击建造时注册
- 加载存档时，地形建筑已注册完成，动态建筑（初始 + 玩家）从存档继续注册剩余空格

### 6.3 地形建筑放置流程（不可拆除）

**在 Godot 编辑器中操作**：

1. 打开地图场景，例如 `modules/world/scenes/forest_map.tscn`
2. 选中 `TerrainBuildings` 节点
3. 右键 -> `Instantiate Child Scene` -> 选择建筑场景（如 `bld_mine.tscn`）
4. 设置 `position`：
   - X = 32 的倍数（对齐网格）
   - Y = `ground_y`（建筑底部贴地面线）
5. 重复放置更多地形建筑

详见 `docs/技术/教程/地图制作教程.md` 步骤 7.2。

### 6.4 初始建筑配置流程（可拆除）

**在 Godot 编辑器中操作**：

1. 打开地图场景，选中 `InitialBuildingsList` 节点
2. 添加子节点或通过 ExportArray 配置每个条目：
   - `def_id`：建筑 ID（如 `bld_house`）
   - `cell_x`：格子坐标（cell_x = world_x / 32）
3. VillageMap._ready() 自动处理首次写入存档

详见 `docs/技术/教程/地图制作教程.md` 步骤 7.3。

### 6.5 玩家建筑放置流程（运行时建造）

**玩家在游戏中操作**（待实现，阶段 0.4+）：

1. 玩家点击"建造"按钮，选择建筑类型
2. 鼠标移动时显示半透明建筑预览
3. PlacementValidator 校验位置是否可放
4. 玩家左键确认：
   - 实例化 `<building_id>.tscn`
   - add_child 到 BuildingHost
   - 注册到 PlacementGrid
   - **写入存档 JSON**（不修改 .tscn/.tres，标记 `is_initial=false`）

### 6.6 存档 JSON 数据结构

存档中的 `buildings` 数组记录动态建筑（初始 + 玩家），**不记录地形建筑**（地形建筑随地图场景打包，不需要存档）：

```json
{
  "version": "0.1.0",
  "save_id": "save_001",
  "maps": {
    "forest_01": {
      "buildings": [
        {
          "instance_id": "dyn_0001",
          "def_id": "bld_house",
          "cell_x": 10,
          "width": 2,
          "health": 200,
          "is_initial": true,
          "built_at_time": 0.0
        },
        {
          "instance_id": "dyn_0002",
          "def_id": "bld_farm",
          "cell_x": 30,
          "width": 2,
          "health": 150,
          "is_initial": false,
          "built_at_time": 12.5
        }
      ]
    }
  }
}
```

`is_initial` 字段区分初始建筑和玩家建筑，便于拆档调试和未来"关卡重置"功能。

### 6.7 拆除建筑的处理

- **地形建筑**：玩家不能拆除（PlacementGrid 标记 `is_terrain=true`，拆除按钮禁用）
- **初始建筑 + 玩家建筑**：可拆除，从存档 JSON 移除，释放 PlacementGrid 格子，退还部分资源

## 七、常见问题

### Q: 为什么游戏内建造不能修改 .tres？
A: `.tres` 是打包进游戏的只读资源。打包后（导出 exe）文件系统只读，运行时修改会失败且数据丢失。建造状态必须存存档。详见 `docs/技术/架构/存储架构设计.md`。

### Q: 为什么所有建筑必须用独立场景文件？
A: 统一架构。三层建筑（地形/初始/玩家）走同一套加载路径，建筑复杂度升级时（从纯 Sprite2D 升级到带动画/血条/PassageBarrier）不需要改加载逻辑。同时与建筑 ID 一一对应，方便 `BuildingRegistry` 索引。

### Q: 三层建筑（地形/初始/玩家）会冲突吗？
A: 不会。三者都通过 PlacementGrid 统一占位登记，地形建筑先注册（地图加载时），动态建筑（初始 + 玩家）后注册（存档加载时），同一格子不会被重复占用。详见上方"六、建筑三层架构"。

### Q: 建筑贴图尺寸怎么定？
A: **贴图尺寸任意**，与占地尺寸解耦。占地宽度 = 条带数 × 32px（决定 PlacementGrid 占用），贴图可以比占地宽或窄。例：一棵树占地 1 格宽（32px 树根），贴图可 96×192px（树冠很宽很高）。Sprite2D 的 `offset.y = -texture_height / 2` 让底部对齐 BuildingRoot 原点。详见 §二·补 贴图与占地解耦。

### Q: 建筑本体能阻挡火柴人通行吗？
A: 能。建筑场景加 `PassageBarrier (Area2D)` 子节点，绘制 CollisionShape2D 矩形标记不可通行区域。雕像上下可绕过、房屋只能从下方过、矿山完全不可过。详见 §二·补 PassageBarrier。

### Q: 如何添加新建筑类型？
A: 1) 编辑 xlsx 添加行；2) 准备美术资源 PNG（任意尺寸）；3) **创建独立场景 .tscn**（命名与 ID 一致，含 PassageBarrier 节点）；4) 导出 .tres。

### Q: 建筑会被火柴人挡住吗？
A: 后续会加 Y-sort（按 Y 坐标排序渲染），火柴人在建筑前方时会遮挡建筑。

### Q: 程序化建筑怎么实现？
A: 架构已就绪。建筑场景根节点可挂 `procedural_building.gd` 脚本，根据 `variant_seed` + `terrain_context` 程序化拼接 sprite 子集。当前 P0 不实现，但建筑场景文件设计为"可挂脚本"就为以后留好了口子。详见架构 §4.1。
