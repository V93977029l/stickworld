### **引言：商业化游戏开发中的“耦合之痛”**

随着游戏项目规模的扩大和团队成员的增加，软件架构的复杂性呈指数级增长。在一个紧密耦合（Tightly Coupled）的系统中，一个微小的改动可能会像多米诺骨牌一样，引发一系列意想不到的连锁反应，导致大量的回归测试、频繁的合并冲突和不断下降的开发效率。对于一个准备商业化的游戏项目而言，这种“牵一发而动周身”的状态是致命的。它不仅拖慢了开发进度，也极大地增加了维护成本和技术债，使得项目难以适应市场变化和玩家需求的快速迭代。

**核心挑战分析：**

1. **高耦合度：** 当项目的不同部分（如玩家、UI、战斗系统）之间存在过多的直接依赖时，修改任何一部分都变得极其困难和危险。例如，UI代码直接引用玩家场景树中的某个具体节点，一旦玩家场景结构调整，UI代码就会崩溃。
2. **多人协作瓶颈：** 在一个多成员的团队中，如果项目结构不清晰，职责划分不明确，开发者们将不可避免地频繁修改同一批文件，尤其是Godot中的场景文件（`.tscn`）。这会导致大量的版本控制合并冲突，将宝贵的开发时间浪费在解决冲突而非创造价值上。
3. **可扩展性与可维护性差：** 一个单体式（Monolithic）的项目结构，使得添加新功能或替换旧系统变得异常痛苦。新功能的代码需要小心翼翼地嵌入到现有复杂的逻辑中，而旧代码因为盘根错节的依赖关系而不敢轻易移除。
4. **迎接AI协作时代：** 未来的游戏开发很可能是人与AI Coding Agent协同工作的模式。如果项目架构混乱、模块边界模糊，AI Agent将难以理解代码库的上下文，无法安全、高效地生成或重构代码，甚至可能在无意中破坏项目结构。

**本报告的目标**，正是为了解决上述挑战，提供一个从架构设计理念、具体文件结构，到团队协作流程，再到AI自动化工作流的端到端解决方案。我们将基于Godot引擎的设计哲学，构建一个既符合人类开发者直觉，又能被AI Agent清晰理解和操作的健壮开发范式。

### **第一部分：Godot模块化架构设计的核心原则**

在设计具体的文件夹结构之前，我们必须首先理解并认同支撑这套架构的底层设计原则。这些原则源于Godot引擎自身的特性以及通用的软件工程最佳实践。

#### **1.1 理解Godot的哲学：拥抱场景树与节点系统**

与其他游戏引擎不同，Godot的设计哲学并不强制开发者遵循经典的软件设计模式，如MVC（模型-视图-控制器）。相反，Godot鼓励开发者“自然地思考游戏”。其核心是**场景（Scene）和节点（Node）** 系统。

- **场景即组件（Scenes as a Complement）：** 在Godot中，几乎万物皆可为场景。一个玩家、一颗子弹、一个UI按钮，都可以是一个独立的场景。更重要的是，场景可以实例化并作为子节点添加到其他场景中，形成一个场景树。这种设计天然地鼓励了“组合优于继承”的原则。我们不应该将一个大型关卡制作成一个包含数千个节点的巨型场景，而应该将其分解为由“玩家”场景、“敌人”场景、“宝箱”场景等多个独立场景组合而成的复合场景。这种组合式的思维是模块化的基石。
- **节点树的内在组织性：** 节点和场景树的父子关系，本身就提供了一种组织和封装游戏对象行为的方式。一个`Player`节点（根）可以拥有`Sprite2D`、`CollisionShape2D`、`Camera2D`等子节点，这些子节点共同构成了“玩家”这个完整的概念。
- **信号：天生的解耦利器：** Godot内置的**信号（Signals）** 机制是其设计的精髓之一。它本质上是观察者模式的一种实现。一个节点可以定义并“发出”（emit）一个信号，而其他任何节点都可以“连接”（connect）到这个信号，并在信号发出时执行相应的回调函数。整个过程中，信号的发出者完全不需要知道谁在监听它，从而实现了完美的解耦。这是我们实现模块间通信、避免直接引用的首选工具。

#### **1.2 耦合的根源与解耦的核心策略**

耦合度描述了模块之间相互依赖的程度。**紧耦合（Tight Coupling）** 意味着模块间关系紧密，修改其一很可能影响其他；**松耦合（Loose Coupling）** 则表示模块相对独立，可以被独立地开发、测试和替换。

**在Godot项目中，常见的耦合来源包括：**

- **硬编码的节点路径 (`get_node("path/to/node")`)：** 这是最常见也是最脆弱的耦合形式。一旦场景树结构发生变化，代码就会立即失效。
- **跨模块的类直接引用：** 模块A的脚本直接实例化或调用模块B的脚本中的具体方法。
- **全局单例（Autoloads）的滥用：** 虽然单例很方便，但如果被用作存储全局状态和进行跨模块调用的“大泥球”，它们就会成为项目耦合的中心枢纽。
- **巨大的、无所不包的场景文件：** 当多个开发者（例如，关卡设计师、程序员、特效师）需要同时修改同一个巨大的关卡场景文件时，版本控制冲突将成为家常便饭。

**我们的核心解耦策略必须围绕以下几点展开：**

1. **优先使用信号：** 模块间的通信应尽可能通过信号进行。例如，当玩家生命值变化时，`Player`模块不应该直接调用`UI`模块的`update_health_bar`方法，而应该发出一个`health_changed(new_health)`信号。`UI`模块自行决定监听此信号并更新显示。
2. **定义清晰的模块接口（Interface）：** 每个模块应该只暴露一小组精心设计的公共方法和信号，作为其与外部世界交互的“契约”。模块内部的实现细节应该被隐藏起来。这遵循了软件工程中的**接口隔离原则（Interface Segregation Principle）**。
3. **依赖倒置：** 高层模块（如游戏逻辑）不应直接依赖于底层模块（如UI、音频播放器）。两者都应该依赖于一个抽象的接口或事件总线。例如，游戏逻辑模块发出一个`play_sound("explosion")`事件，而不是直接调用`AudioManager.play_explosion_sound()`。
4. **物理隔离：** 将每个模块相关的所有文件（场景、脚本、资源）都放在同一个文件夹内。这种物理上的内聚性，是实现逻辑上低耦合的第一步。

#### **1.3 文件夹结构的组织哲学：按功能而非类型**

许多开发者，特别是从Unity等引擎转型而来的，习惯于按资源类型组织文件夹，例如：

```
/scenes
/scripts
/materials
/textures
/audio
```

这种结构在项目初期看似整洁，但随着项目规模的扩大，其弊端会愈发明显。当你想找到与“玩家”相关的所有东西时，你需要在`scenes`、`scripts`、`textures`等多个文件夹之间来回跳转，这极大地增加了认知负荷，并且丢失了文件之间的上下文关联。这种组织方式本质上是反模块化的。

我们必须采用**按功能（By Feature）**或**按上下文（By Context）** 的组织方式。这意味着，与同一游戏功能（如玩家、敌人、库存系统）相关的所有文件都应该被放在同一个专属文件夹中。

**按功能组织的好处是显而易见的：**

- **高内聚性：** 相关的文件物理上聚集在一起，便于查找和理解。
- **模块独立性：** 每个功能文件夹都可以被视为一个独立的“微型项目”。理论上，你可以将整个`/player`文件夹复制到另一个项目中，稍作调整即可复用。
- **降低协作冲突：** 如果美术师在修改玩家的贴图，程序员在修改玩家的控制逻辑，他们都只在`/player`文件夹内工作，减少了与其他模块开发人员的交叉影响。
- **清晰的职责划分：** 可以将整个功能文件夹的所有权分配给某个开发者或小组，权责明确。

### **第二部分：商业级Godot项目的模块化文件夹结构方案**

基于上述原则，我们设计了一套专为商业级项目和多人协作优化的模块化文件夹结构。这套结构旨在实现清晰的模块划分、强制的边界以及高度的可维护性。

#### **2.1 顶层目录结构设计**

这是项目资源路径`res://`（本项目根目录下的transparent-pet文件夹才是Godot的根目录，其他文件夹都是母项目的其他模块，比如GDExtension插件代码）的根目录结构。每个文件夹都有其明确的用途。

```
/ (res://)
|
├── core/                  # 核心系统与基础设施 (Core Systems & Infrastructure)
├── modules/               # 游戏功能模块 (Gameplay Feature Modules)
├── assets/                # 全局共享资源 (Global Shared Assets)
├── addons/                # 第三方插件 (Third-party Addons)
├── prototypes/            # 原型与实验区 (Prototypes & Sandbox)
├── tools/                 # 自定义编辑器工具 (Custom Editor Tools)
├── tests/                 # 自动化测试脚本 (Automated Tests)
└── docs/                  # 项目文档 (Project Documentation)
```

**各顶层目录职责说明：**

- **`core/`**: 存放项目的根基，是所有其他模块依赖的基础设施。这里的代码应具备高度的稳定性和通用性，修改前需要经过严格的评审。可以将其视为游戏的“操作系统”。
- **`modules/`**: 项目的心脏。所有独立的游戏玩法功能，如玩家系统、战斗系统、任务系统等，都以子文件夹的形式存放在这里。这是开发活动最频繁的区域。
- **`assets/`**: 存放那些被多个模块共享的、不属于任何特定模块的通用资源。例如，通用的UI图标、全局着色器、项目主字体等。
- **`addons/`**: 用于存放从Godot Asset Library或第三方获取的插件和工具。保持这个文件夹的独立性便于管理和更新外部依赖。
- **`prototypes/`**: 一个“沙盒”区域，供开发者进行快速原型设计和功能实验，而不会污染主项目结构。这里的文件不应被正式的游戏逻辑所依赖。
- **`tools/`**: 存放为提升开发效率而编写的自定义编辑器脚本和工具（`@tool` 脚本）。
- **`tests/`**: 存放所有自动化测试脚本。一个良好的实践是，其内部结构镜像`core/`和`modules/`的结构，便于定位测试对象。
- **`docs/`**: 存放项目的设计文档、技术文档、美术规范等。

#### **2.2 核心模块 (`core/`) 详解**

`core/` 模块的目标是提供稳定、可靠的底层服务，并让上层的游戏功能模块与之解耦。

```
core/
|
├── autoload/              # 全局单例/服务
|   ├── event_bus.gd       # 全局事件总线
|   ├── scene_manager.tscn # 场景加载与过渡管理器
|   ├── save_manager.gd    # 存档/读档服务
|   └── config_manager.gd    # 游戏配置管理
|
├── ui_framework/          # UI基础设施
|   ├── base_screen.gd     # 所有UI界面的基类
|   ├── components/        # 通用UI组件 (如自定义按钮, 弹窗)
|   └── theme/             # 全局UI主题 (theme.tres)
|
├── services/              # 抽象服务接口与实现
|   ├── analytics/         # 数据分析服务
|   ├── iap/               # 内购服务
|   └── audio_manager.gd   # 音频管理器 (通常也是单例)
|
└── utils/                 # 通用工具类
    ├── math_utils.gd      # 数学帮助函数
    └── node_utils.gd      # 节点操作帮助函数
```

**关键设计点：**

- **`event_bus.gd`**: 这是一个至关重要的单例，它实现了发布-订阅模式。模块之间不直接通信，而是向`EventBus`发布事件（例如 `EventBus.publish("player_died")`），或者订阅`EventBus`上的事件（`EventBus.subscribe("player_died", on_player_died)`）。这极大地降低了模块间的耦合。
- **服务抽象化**: 对于像内购、分析这类服务，`core/` 中应定义其接口，具体的实现可以根据平台（Steam, Mobile）不同而替换，上层模块只与接口交互。

#### **2.3 游戏功能模块 (`modules/`) 设计**

这是体现模块化思想的核心。每个子文件夹都是一个垂直切片，包含了一个完整功能所需的一切。

**以** **`player`** **模块（虽然当前项目是个桌宠项目，没有player角色）为例：**

```
modules/
|
└── player/
    |
    ├── scenes/                    # 模块相关的场景
    |   ├── player.tscn            # 玩家主场景 (根节点)
    |   └── components/            # 玩家的组件化场景
    |       ├── health_component.tscn
    |       └── movement_component.tscn
    |
    ├── scripts/                   # 模块相关的脚本
    |   ├── player_controller.gd   # 玩家主逻辑
    |   ├── health_component.gd
    |   └── movement_component.gd
    |
    ├── assets/                    # 模块专属的资源
    |   ├── models/                # 3D模型
    |   ├── textures/              # 贴图
    |   ├── animations/            # 动画 (player.anim)
    |   └── sfx/                   # 音效
    |
    ├── ui/                        # 模块专属的UI
    |   ├── player_hud.tscn        # 玩家的平视显示器 (如血条)
    |   └── player_hud.gd
    |
    └── api.gd                     # 模块的公共接口契约 (关键！)
```

**关键设计点：**

- **自包含性**: `player/` 文件夹包含了实现玩家功能所需的一切。如果想在另一个游戏中复用这个玩家，理论上只需拷贝整个文件夹。
- **组件化场景**: `player.tscn` 场景本身可能很简单，其核心功能由多个子场景（如`health_component.tscn`）组合而成。这种细粒度的场景划分，极大地降低了多人编辑同一个场景文件导致冲突的概率。
- **`api.gd`** **- 模块接口契约**: 这是一个可选但强烈推荐的实践。它是一个脚本（通常附加在模块的根场景节点上，或者作为一个静态类），明确定义了该模块对外暴露的公共接口。
  - **公共方法**: 如 `take_damage(amount)`。
  - **公共信号**: 如 `signal health_changed(current_health, max_health)`。
  - **公共属性**: 如 `var is_invincible: bool`。
    所有其他模块与`player`模块的交互，**必须且只能**通过`api.gd`中定义的接口进行。这为模块间建立了一道清晰的防火墙，也为未来的AI Agent代码生成提供了可解析的契约。

**以RPG游戏为例，游戏必要模块文件夹：**

```
modules/
|
├── player/                 # 玩家角色、控制、状态等
├── inventory/              # 背包系统、物品管理
├── combat/                 # 战斗系统 (技能、伤害计算、效果)
├── dialogue/               # 对话系统
├── quest/                  # 任务系统
├── world_map/              # 世界地图、导航
├── enemy_ai/               # 敌人AI (巡逻、追击、攻击)
├── level_management/       # 关卡管理、场景加载
├── crafting/               # 制造/合成系统
└── shop/                   # 商店/交易系统
```

#### **2.4 命名规范与项目配置**

- **项目配置文件 (`project.godot`)**:
    - 这是一个高冲突风险区，因为它包含了所有人的输入映射、单例加载、渲染设置等。
    - **解耦配置**: 尽可能将配置信息从`project.godot`中移出，放到自定义的资源文件（`.tres`）或配置文件（如`.json`, `.cfg`）中，这些文件可以存放在各自的模块内。例如，每个模块的输入动作可以定义在自己的配置文件中，由一个核心的`InputManager`在启动时加载。

### **第三部分：面向Coding Agent的自动化工作流搭建**

未来的开发团队将是人机协同的。一个设计良好的模块化架构，是AI Coding Agent能够安全、高效工作的先决条件。我们的工作流需要进一步演进，以适应这种新的生产力范式。

#### **3.1 Coding Agent工作流概述**

这个工作流旨在将AI Agent从一个简单的代码补全工具（Copilot）转变为一个能够独立完成开发任务的自主代理（Autopilot）。

**工作流闭环：**

1. **任务定义 (Prompting)**: 产品经理或开发者用自然语言清晰地定义一个开发任务。
2. **Agent代码生成 (Generation)**: AI Agent解析任务，理解项目架构（通过学习我们的设计文档和模块`api.gd`），在指定的模块内生成新的代码、场景和资源。
3. **自动化验证 (Validation)**: Agent生成的代码被自动推送到一个特性分支，并触发CI/CD（持续集成/持续部署）流水线。流水线会自动执行编译、单元测试、接口契约检查和场景结构检查。
4. **人类审查 (Review)**: 只有通过所有自动化检查的代码，才会生成一个合并请求，并通知相关的人类开发者进行最终的逻辑审查。
5. **集成 (Integration)**: 人类开发者批准后，代码被合并到`develop`分支。

#### **3.2 自动化测试与契约验证**

这是确保AI Agent不会"好心办坏事"的关键安全网。我们需要建立一个强大的自动化验证流水线，最好使用GitHub Actions等工具。

**流水线中的关键步骤：**

1. **单元测试 (Unit Testing)**:
   - 我们约定所有核心逻辑都必须有配套的单元测试。AI Agent在生成功能代码时，也必须同时生成测试用例。
   - 使用像 `GdUnit4` 或 `godot-tester` 这样的测试框架。
   - 流水线会在每次提交时自动运行所有测试，确保新代码没有破坏现有功能。
2. **场景层次结构验证 (Scene Hierarchy Validation)**:
   - AI Agent可能会在修改`.tscn`文件时犯错，例如删除了一个必要的子节点，或更改了节点类型。
   - 我们可以在流水线中加入一个步骤，使用Godot的无头模式（headless mode）运行一个自定义的验证脚本。
   - 这个脚本可以加载AI修改过的场景，并断言其结构是否符合预期。例如，检查`player.tscn`是否仍然拥有一个`CollisionShape2D`子节点，并且其根节点类型是否为`CharacterBody2D`。

## Git 分支与工作流规范

### 分支角色

| 分支 | 用途 | 寿命 |
|------|------|------|
| `main` | 稳定发布分支，CI 自动测试+构建 | 永久 |
| `dev/xxx` | 各开发者的个人集成分支（如 `dev/fan`） | 永久 |
| `agent/xxx` | AI 执行任务的功能分支 | **合并后删除** |

### AI 工作流（每次任务的标准流程）

1. **确定当前开发分支** — 先看用户当前在哪个 `dev/xxx` 分支上，所有操作围绕此分支展开
2. **创建功能分支** — `git checkout -b agent/xxx <当前dev分支>`
3. **编写代码+测试** — 修改 Godot 项目 → `godot --headless` 跑测试 → 通过后原子化提交
4. **推送功能分支** — `git push -u origin agent/xxx`
5. **合并回 dev 分支** — `git checkout <当前dev分支> && git merge agent/xxx && git push`
6. **自动清理 agent 分支** — AI 全权管理所有 `agent/*` 分支的生命周期，合并后立即删除（本地+远程）：
   ```
   git branch -d agent/xxx
   git push origin --delete agent/xxx
   ```
7. **回到 dev 分支** — 继续下一个任务

### 合并到 main

当 dev 分支积累了一批经过测试的改动，由开发者发起或 AI 执行：
```
git checkout main
git pull origin main
git merge <当前dev分支> --no-edit
git push origin main
git checkout <当前dev分支>
```

### AI 的提交规范

- 每次提交是原子化的最小功能单元
- 提交信息用中文，格式：`类型(模块): 描述`
  - 类型: `fix` / `feat` / `refactor` / `test` / `docs` / `chore`
  - 示例: `fix(ci): 测试失败阻塞构建` `test(core): 添加 config_manager 单元测试`
- **每次修改 Godot 项目文件后必须用 `godot --headless` 验证编译通过再提交**

### 待办项维护

- 架构/工程改进待办项统一记录在 `docs/架构改进待办项.md`
- **每完成一项改进，必须从该文档中删除对应条目**，保持文档反映真实剩余工作
- 新增改进项时追加到文档末尾对应优先级分区
